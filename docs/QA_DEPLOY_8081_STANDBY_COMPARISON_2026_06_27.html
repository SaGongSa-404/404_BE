QA 배포 방식 비교: 기존 재시작 vs 8081 스탠바이

이 문서는 QA 서버 배포에서 8081 스탠바이 절차를 사용할 때와 사용하지 않을 때의 사용자 영향, 배포 소요 시간, 운영 장단점을 비교한다.

기존 방식
짧은 배포, 노출되는 끊김

8080 서비스를 바로 재시작하므로 배포 단계는 단순하지만, 기동 완료 전까지 외부 요청은 502 또는 연결 실패가 될 수 있다.

8081 방식
긴 배포, 끊김 최소화

후보 jar를 8081에서 먼저 검증하고 Caddy를 전환하므로 작업 시간은 늘지만 사용자가 보는 다운타임을 줄인다.

권장 판단
8081 스탠바이 유지

비공개 테스트 기간에는 배포 속도보다 외부 사용자의 요청 성공률과 복구 여지가 더 중요하다.

비교 기준

현재 QA VM은 2 vCPU, 약 4GB 메모리로 업그레이드되었고, Caddy가 localhost:8080 또는 localhost:8081로 reverse proxy를 전환하는 구조다. 8081 절차는 완전한 로드밸런서 blue/green은 아니지만, 단일 VM에서 가능한 경량 무중단 배포에 가깝다.
항목 기존 8080 직접 재시작 8081 스탠바이 절차
배포 흐름 jar 업로드 후 wigul-backend 8080 서비스를 바로 재시작 candidate jar를 8081로 먼저 기동, health 성공 후 Caddy 전환, 8080 교체 후 복귀
사용자 체감 끊김 발생 가능. 8080 재시작 동안 Caddy upstream이 비어 502가 날 수 있음 최소화. 8080 재시작 중 외부 요청은 8081이 받음
배포 작업 시간 더 짧음. standby 기동과 Caddy 왕복 전환이 없음 더 김. 8081 기동 대기와 8080 재기동 대기를 모두 수행
실패 시 상태 8080이 실패하면 외부 요청도 실패할 가능성이 큼 8080 복구 실패 시 Caddy를 8081에 남겨 서비스 지속 가능
복잡도 낮음. systemd restart 중심 중간. standby pid, Caddyfile 전환, rollback guard가 필요
리소스 사용 Java 프로세스 1개 배포 중 잠시 Java 프로세스 2개. 4GB 업그레이드 이후 허용 가능한 수준

시간 차이

시간은 “전체 GitHub Actions 실행 시간”과 “사용자에게 노출되는 끊김 시간”을 나눠서 봐야 한다. 8081 방식은 전체 배포 시간은 늘지만, 사용자에게 노출되는 실패 시간을 줄이는 방식이다.

구분 기존 방식 예상 8081 방식 예상 판단
추가 준비 시간 거의 없음 8081 standby 기동 대기 약 20~40초 수준 8081 방식이 더 오래 걸림
8080 재시작 시간 현재 4GB 기준 약 20~40초 수준으로 예상 동일하게 발생하지만 외부 트래픽은 8081이 받음 작업 시간은 비슷하나 사용자 영향이 다름
Caddy 전환 시간 없음 8081 전환과 8080 복귀가 각각 짧게 발생 초 단위 추가 비용
사용자 체감 실패 시간 8080 기동 전까지 502 가능 정상 경로에서는 거의 없음. 실패 시에도 8081 유지 가능 8081 방식 우위
총 배포 시간 더 짧음 대략 8081 기동 대기만큼 더 길어짐 속도보다 안정성 우선이면 수용 가능

8081 절차를 안 했을 때

장점

  • workflow가 짧고 단순하다.
  • 배포 총 시간이 더 짧다.
  • 추가 프로세스와 pid 관리가 없다.

단점

  • 재시작 구간에 외부 사용자가 502를 볼 수 있다.
  • 기동이 늦어지면 장애처럼 보이는 시간이 길어진다.
  • 비공개 테스트 중 사용자 행동과 겹치면 신뢰도가 떨어질 수 있다.

8081 절차를 했을 때

장점

  • 새 jar가 8081에서 뜨는지 먼저 확인하고 외부 전환한다.
  • 8080 재시작 동안에도 외부 요청을 받을 수 있다.
  • 8080 복구 실패 시 Caddy를 8081에 남겨 서비스 유지가 가능하다.
  • 비공개 테스트 기간에 “배포 때문에 터졌다”는 리스크를 줄인다.

단점

  • 전체 배포 시간은 더 길어진다.
  • Caddyfile 전환, standby 종료, 실패 시 복구 규칙이 추가된다.
  • 배포 중 잠시 Java 프로세스 2개가 떠서 메모리를 더 쓴다.

운영 판단

QA 비공개 테스트 기준에서는 8081 스탠바이 절차를 사용하는 편이 더 낫다. 이유는 배포 총 시간이 조금 늘어나는 비용보다, 실제 사용자 요청이 실패로 보이는 시간을 줄이는 이득이 더 크기 때문이다.

특히 테스트 인원이 동시에 앱을 쓰는 상황에서는 20~40초의 재시작 공백도 로그인, 알림, 위시리스트, 홈 요약 같은 API 실패로 체감될 수 있다. 8081 절차는 이 구간을 standby로 흡수하고, 실패 시에도 8081을 임시 운영 상태로 남겨 복구 시간을 벌 수 있다.

채택 조건

조건 필요 이유 현재 판단
QA VM 메모리 여유 배포 중 Java 프로세스 2개가 잠시 떠야 함 충족. 4GB 업그레이드 후 여유 확인
Caddy 전환 권한 workflow가 Caddyfile을 8081/8080으로 바꾸고 reload해야 함 PR 적용 후 첫 배포에서 확인 필요
8081 포트 비어 있음 standby 프로세스 기동 필요 사전 검사 포함
DB migration 호환성 8080과 8081이 짧게 같은 DB를 바라볼 수 있음 파괴적 migration은 별도 배포 계획 필요
