# 워크플로 개선 버전

작성일: 2026-03-19  
기준 문서: `reports/workflow_review_for_agentB_2026-03-19.md`

이 문서는 평가 A의 지적을 바탕으로 워크플로를 더 작고, 더 명확하고, 더 유지보수 가능하게 고친 버전이다.

## 1. 개선 방향 요약

기존 문제:
- 파일이 많아질수록 기준 문서가 흐려짐
- handoff 형식이 완전히 표준화되지 않음
- 아이디어가 쉽게 폐기되지 않음
- 회차가 늘수록 중복 기록이 늘어남

개선 원칙:
1. `소스 오브 트루스는 항상 1개`
2. `에이전트별 로그는 참고용`
3. `회차가 늘수록 기록은 더 짧아져야 함`
4. `살릴 기준`보다 `죽일 기준`을 먼저 명시
5. `취합`은 요약이 아니라 결정과 다음 액션 중심

## 2. 새 워크플로 구조

### A. 핵심 루프

반드시 아래 4단계만 수행:
1. 1번 에이전트: 원안
2. 2번 에이전트: 검증
3. 3번 에이전트: 전략 보강
4. 1번 에이전트: 수정안

규칙:
- 1회차는 아이디어 발굴
- 2회차는 wedge 축소
- 3회차는 포지셔닝 고정

### B. 확장 루프

3회차 이후 4~10회차는 전체 재서술을 금지하고 아래만 기록:
- 이번 회차에 더 좁힌 것
- 이번 회차에 버린 것
- 새로 생긴 리스크
- 유지되는 강점
- 다음 회차 진입 여부

### C. 후속 제품화

상위 3개만 아래 단계로 이동:
1. 점수표 재산정
2. 사업성 비교표
3. PRD
4. MVP 화면 구조
5. 인터뷰/파일럿 운영안

## 3. 문서 구조 개선안

### 반드시 존재해야 하는 파일

1. `master_report.md`
   - 현재 라운드의 기준 문서
   - 사용자가 먼저 읽는 파일

2. `agent_logs/`
   - 1번/2번/3번 에이전트의 참고 로그

3. `scorecards/`
   - 점수표 재산정 문서

4. `productization/`
   - PRD, MVP 화면 구조, 인터뷰안

### 권장 폴더 구조

```text
reports/
  current/
    master_report.md
    scorecard.md
  rounds/
    round1/
    round2/
    round3/
  agent_logs/
    agent1.md
    agent2.md
    agent3.md
  productization/
    top3_comparison.md
    prd_idea1.md
```

## 4. 소스 오브 트루스 규칙

- 매 라운드마다 기준 문서는 하나만 둔다.
- 기준 문서 외 파일은 모두 `참고 로그`로 취급한다.
- 취합 에이전트는 기준 문서를 갱신하되, 이전 버전은 아카이브로 이동한다.

권장 규칙:
- 사용자가 먼저 읽는 파일은 항상 `current/master_report.md`
- `roundX` 파일은 회차별 백업
- `agent_logs`는 검수용

## 5. handoff 템플릿 고정안

### 1번 -> 2번

```text
- 아이디어명
- 한 문장 pain
- 구매자
- 실사용자
- 반복 사용 트리거
- 돈을 낼 이유
- 이번 회차에 버린 것
- 6주 파일럿 MVP
```

### 2번 -> 3번

```text
- 유사 서비스/대체재
- 수요가 강한 근거
- 돈을 낼 이유
- 돈을 안 낼 이유
- 치명적 리스크 3개
- 생존 판정
- 살리는 조건 또는 폐기 이유
```

### 3번 -> 1번

```text
- 유지 강점
- 버릴 요소
- 차별화 포인트
- 첫 고객 채널
- 6주 KPI
- 수익화 방향
- 1번이 반드시 답해야 할 질문 2개
```

## 6. 폐기 기준 강화안

아래 중 2개 이상이면 기본적으로 `보류 또는 폐기`:
- 반복 사용 빈도가 낮음
- 대체재 대비 wedge가 약함
- 돈을 낼 이유보다 안 낼 이유가 더 강함
- 도입 속도가 너무 느림
- 세일즈 사이클이 지나치게 무거움
- 실사용자가 새 입력을 거의 하지 않음
- 독립 제품보다 기존 시스템 부가기능처럼 보임

즉시 폐기 조건:
- pain은 있지만 독립 구매 구조가 전혀 없음
- 규제/정확도 책임이 MVP 범위를 넘김
- 기존 상위 아이디어와 지나치게 중복됨

## 7. 에이전트별 수정 규칙

### 1번 에이전트
- 반드시 `지킨 것 / 버린 것 / 더 좁힌 것`을 쓴다.
- 새 기능 추가보다 범위 제거를 우선한다.
- 6주 파일럿 밖의 기능은 자동 보류한다.

### 2번 에이전트
- `살림 / 조건부 살림 / 보류 / 폐기`를 반드시 적는다.
- `안 사는 이유`를 `사는 이유`와 같은 비중으로 적는다.
- 한국 로컬 미스매치와 글로벌 대체재를 구분해서 본다.

### 3번 에이전트
- `버릴 요소`를 먼저 쓴다.
- 첫 고객 채널은 최대 1~2개만 남긴다.
- KPI는 정량으로 쓴다.

### 취합 에이전트
- 재요약보다 `무엇을 남기고 무엇을 버릴지`를 결정한다.
- 최종 문서는 3개 질문에 답해야 한다:
  1) 무엇이 가장 유망한가
  2) 왜 그런가
  3) 다음 액션은 무엇인가

## 8. 회차가 늘어날 때의 기록 규칙

1~3회차:
- 전체 내용을 비교적 자세히 기록

4~10회차:
- 변화점만 기록
- 이전 회차와 같은 설명 반복 금지
- 한 회차당 아이디어별 5줄 내외 유지 권장

## 9. 사용자 읽기 순서 개선안

사용자는 항상 아래 순서로 본다.

1. `current/master_report.md`
2. `current/scorecard.md`
3. 필요하면 `agent_logs/agent1.md`, `agent2.md`, `agent3.md`
4. top3만 `productization/` 문서 확인

## 10. 최종 권장 운영 문장

`기준 문서는 하나만 유지하고, 에이전트는 역할별로 분리하되, handoff를 완전히 표준화하며, 회차가 늘수록 기록은 짧아지고 폐기 기준은 더 엄격해져야 한다.`
