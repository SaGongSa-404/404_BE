# 아이디어 멀티에이전트 3회 루프 완결 보고서

작성일: 2026-03-18  
전제: 한국 시장에서 먼저 시작하고, 글로벌 SaaS/워크플로우 제품으로 확장 가능한 소프트웨어 아이디어를 우선 검토했다.  
루프 규칙: 각 아이디어는 `1번 원안 -> 2번 검증 -> 3번 보강 -> 1번 수정안` 흐름을 총 3회 반복했다.  
목표: 반복 사용성, 명확한 수요층, 지불 의사, 한국 현장 적합성, 빠른 파일럿 가능성을 모두 보는 것.

## 전체 결론

- 3회 루프까지 돌린 뒤에도 가장 강하게 남은 후보는 `ExportDoc Copilot`, `GlobalWorker Desk`, `HakwonOps`, `ShiftGuard for Franchises`, `PostShip Recovery`였다.
- 공통점은 `반복 빈도가 높고`, `실수/누락의 비용이 크며`, `엑셀/카톡/전화의 비공식 운영이 아직 큰 시장`이라는 점이었다.
- 3회차까지 갈수록 살아남은 방향은 거의 모두 `올인원`이 아니라 `하나의 반복 워크플로우를 확실히 개선하는 도구`였다.

## 아이디어별 3회 루프 결과

### 1. ExportDoc Copilot

`1회차 원안`
- 한국 중소 수출기업의 서류, 원산지, 바이어 요청, 포워더 협업이 사람 경험에 과도하게 의존한다.
- 타겟은 연 매출 30억~500억 규모의 제조/유통 수출 중소기업 운영팀과 무역실무 담당자였다.

`1회차 검증`
- KTNET uTradeHub, 기존 무역 ERP, 포워더 포털이 이미 존재해 니즈 자체는 검증돼 있었다.
- 진짜 문제는 시스템 부재가 아니라 현장 예외처리 UX 부재였다. 수요 강도는 `강함`.

`1회차 보강`
- 올인원 무역 툴이 아니라 `서류 예외처리 레이어`로 들어가야 한다는 결론이 나왔다.
- 첫 세그먼트는 화장품/식품/산업재처럼 서류 요구사항이 잦은 중소 수출사로 좁혔다.

`1회차 수정안`
- 이메일/엑셀 업로드, 출하별 체크리스트 자동 생성, 누락 경고, 진행상태 보드 중심 MVP로 축소했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `미국/동남아 수출 비중이 높은 화장품·식품 브랜드`에 더 집중했다.
- 새 가설은 “고객이 원하는 것은 문서 자동작성보다도 바이어/포워더/실무자 사이의 누락 방지”였다.
- 달라진 점:
  - 품목 범위를 줄였다.
  - 이메일 기반 협업을 핵심 입력 채널로 고정했다.
  - 내부 ERP 대체가 아니라 외부 커뮤니케이션 정합성에 집중했다.
- 난이도 변화:
  - HS코드/규정 자동판단 비중을 낮춰 구현 난도는 내려갔다.
  - 메일 파싱과 문서 비교 정확도는 오히려 중요해졌다.

`2회차 검증`
- Descartes Visual Compliance, Shipping Solutions, Freightos 계열 사례를 볼 때 수출 보조 시장은 이미 검증돼 있었다.
- 하지만 고객은 “규정 DB”보다 “실무 누락 방지”에 더 빨리 돈을 낼 가능성이 컸다.
- 치명 리스크:
  - 국가별 규정 최신성 책임을 제품이 과하게 떠안을 수 있다.
  - 품목을 넓히면 바로 정확도와 CS 비용이 폭증한다.

`2회차 보강`
- 차별화 포인트를 `메일 기반 바이어 요구사항 추출`, `선적 건별 누락 경고`, `포워더 협업 로그`로 재설계했다.
- 수요층을 `무역팀 1~3명 체제의 중소 수출사`로 더 좁혔다.
- 초기 수요 확보는 관세사/포워더 제휴보다도 “최근 반려 사례 진단 리포트”형 세일즈가 더 유효하다고 판단했다.

`2회차 수정안`
- MVP를 `바이어 메일 업로드 -> 선적건 생성 -> 문서 체크리스트 생성 -> 누락/불일치 경고 -> 포워더 공유` 흐름으로 고정했다.
- 가격은 기업당 월 구독 + 월 선적 건수 구간 과금이 가장 자연스럽다고 정리했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 `서류 작업 도우미`에서 `선적 실패 비용 절감 OS`로 포지셔닝을 다시 잡았다.
- 새 가설은 “고객은 자동화보다 통관 지연과 바이어 컴플레인을 줄여주는 시스템에 반응한다”였다.
- 달라진 점:
  - 사용자 가치를 시간 절약에서 손실 방지로 바꿨다.
  - 내부 기능 설명보다 위험 점수와 사전 경고 흐름을 강화했다.
  - 고객 성공 지표를 문서 작성 속도에서 선적 지연/반려 감소로 바꿨다.
- 난이도 변화:
  - 규정 자동판단 대신 룰 기반 경고를 채택해 MVP 난이도는 다시 낮아졌다.
  - 반면 품목별 템플릿 설계 중요도는 높아졌다.

`3회차 검증`
- 이미 있는 솔루션과 정면승부하려면 범용성보다 `한국 수출실무자용 예외처리 UX`가 더 중요했다.
- 강한 수요는 확인됐고, 진입 전략만 충분히 좁히면 실제 사용 빈도도 높을 것으로 봤다.

`3회차 보강`
- 최종 차별화는 `품목 특화 템플릿`, `바이어 메일-문서-포워더 전달의 한 화면 연결`, `반려/지연 위험 점수`로 정리했다.
- 초기 고객은 미국 수출 화장품/식품 업체 10곳 내외 파일럿이 최적이라고 판단했다.

`3회차 수정안`
- 최종 MVP:
  - 바이어 메일/문서 업로드
  - 선적 건별 체크리스트
  - 누락/불일치 경고
  - 포워더 공유용 상태보드
  - 반려/지연 사후분석
- 최종 평가: `매우 강함`
- 이유: 반복성이 매우 높고, 실패 비용이 커서 구독료를 설명하기 쉽다.

### 2. HakwonOps

`1회차 원안`
- 학원 운영의 출결, 보강, 상담, 결제, 학부모 커뮤니케이션이 분절돼 있다.
- 타겟은 중소형 영어/수학/입시 학원, 교습소, 지역 체인형 학원이었다.

`1회차 검증`
- 학원 관리 솔루션과 출결 앱은 있지만 현장에서는 카톡/전화/엑셀이 여전히 강했다.
- 사교육비와 참여율을 고려하면 수요 자체는 `매우 강함`이었다.

`1회차 보강`
- LMS나 AI 튜터가 아니라 `학부모 신뢰 운영`에 집중해야 한다는 방향이 나왔다.
- 미수금, 퇴원 징후, 상담 히스토리를 한 흐름으로 묶는 것이 핵심이었다.

`1회차 수정안`
- 출결, 보강 일정, 미수금 관리, 상담 로그, 학부모 알림, 퇴원 위험 신호를 MVP 축으로 삼았다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `초중등 영어/수학 보습학원`에만 집중했다.
- 새 가설은 “학원이 돈을 내는 이유는 운영 편의보다도 학부모 불안 감소와 재등록률 개선”이었다.
- 달라진 점:
  - 입시/예체능 등 넓은 범위를 버렸다.
  - 상담과 보강보다 `학부모 신뢰 브리핑`을 전면에 세웠다.
  - 원장용 대시보드보다 실장/행정이 매일 쓰는 흐름으로 재배치했다.
- 난이도 변화:
  - 콘텐츠 기능을 버려 개발 난도는 크게 내려갔다.
  - 대신 카톡형 메시지 UX와 자동 브리핑 문구 품질이 중요해졌다.

`2회차 검증`
- BAND, 학원 관리 솔루션, 자체 문자 시스템이 대체재였지만, 아직도 운영 파편화가 심했다.
- 고객은 “더 많은 기능”보다 “민원 감소와 재등록 방지”를 원할 가능성이 컸다.

`2회차 보강`
- 차별화 포인트를 `수업 후 학부모 브리핑 자동화`, `결석/미수금/상담 이력 연결`, `퇴원 위험 신호`로 구체화했다.
- 초기 수요층은 원장 1명 + 행정 1~3명 체제의 학원으로 다시 좁혔다.

`2회차 수정안`
- 핵심 사용자 플로우를 `학생 등록 -> 출결/숙제/시험 이벤트 축적 -> 학부모 브리핑 자동 생성 -> 상담/보강 조정 -> 퇴원 위험 경고`로 재정의했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품 정체성을 `학원 운영 도구`에서 `학부모 신뢰 유지 OS`로 더 날카롭게 바꿨다.
- 새 가설은 “재등록을 잃는 핵심 이유는 성적 자체보다도 학부모가 느끼는 불투명함”이었다.
- 달라진 점:
  - 출결 관리보다 브리핑 품질과 상담 이력 연결을 앞세웠다.
  - 선생님 입력 부담을 줄이는 자동 요약에 투자했다.
  - 수익모델을 학생 수 과금보다 운영 인원 수 + 알림 사용량 기준으로 단순화했다.
- 난이도 변화:
  - 복잡한 교육 콘텐츠 기능을 버려 난도는 낮아졌다.
  - 상담 기록 구조화와 자동 요약 로직은 중요도가 높아졌다.

`3회차 검증`
- 이미 플레이어가 있음에도 불구하고 `학부모 신뢰 유지`라는 세일즈 메시지는 여전히 비어 있는 공간이 있었다.
- 한국형 카톡/문자 기반 흐름과 원장-실장-학부모 관계를 잘 반영하면 실제 사용성은 높을 것으로 봤다.

`3회차 보강`
- 최종 차별화는 `학부모용 짧은 브리핑`, `퇴원 위험 신호`, `미수금/보강/상담의 한 화면 연결`로 수렴했다.
- 초기 고객 유입은 지역 학원장 커뮤니티, 학원 컨설턴트, 학원 프랜차이즈 본부 제휴가 적합했다.

`3회차 수정안`
- 최종 MVP:
  - 출결/숙제/시험 업로드
  - 학부모 브리핑 자동 생성
  - 상담/보강/미수금 연동
  - 퇴원 위험 알림
  - 월말 신뢰 리포트
- 최종 평가: `강함`
- 이유: 반복 사용성이 높고, 한국형 운영 미스매치가 커서 니치하지만 깊게 들어갈 수 있다.

### 3. ShiftGuard for Franchises

`1회차 원안`
- 다점포 카페/식당의 시프트, 근태, 급여 기초자료, 주휴/연장 이슈가 현장마다 엉켜 있다.
- 타겟은 5~100개 매장 규모의 프랜차이즈 운영팀이었다.

`1회차 검증`
- 근태 앱과 POS 부가기능은 많지만, 인건비율과 예외처리 연결은 여전히 수작업 비중이 컸다.
- 수요 강도는 `강함`으로 평가됐다.

`1회차 보강`
- 급여 실행까지 다 하지 말고 `시프트 + 리스크 방지` 중심으로 좁혀야 한다는 방향이 나왔다.

`1회차 수정안`
- 시프트 작성, 근무 변경 요청, 주휴/연장 경고, 인건비 대시보드를 MVP로 설정했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `10~50개 매장 규모의 카페/치킨/버거 브랜드 운영팀`에만 집중했다.
- 새 가설은 “점주보다 본사가 먼저 돈을 내는 이유는 편의성보다도 노동 리스크와 인건비율 관리”라는 것이었다.
- 달라진 점:
  - 일반 스케줄링에서 본사용 가드레일로 초점을 이동했다.
  - 매장 단위보다 본사 표준 룰 배포 기능을 강화했다.
  - 채용/급여 실행 범위를 더 후순위로 밀었다.
- 난이도 변화:
  - 급여 지급 연동을 빼서 초기 구현 난도는 낮아졌다.
  - 대신 POS 매출 기반 권장 시프트 계산 로직 중요도는 올라갔다.

`2회차 검증`
- 7shifts, Restaurant365, 국내 POS 부가기능 등 유사군은 많지만 본사형 한국 근로 리스크 대응은 여전히 틈이 있었다.
- 경쟁은 있으나 반복 사용성과 ROI가 뚜렷해 시장성은 유지됐다.

`2회차 보강`
- 차별화 포인트를 `본사 룰 강제`, `주휴/연장 사전경고`, `점포별 인건비율 비교`로 정리했다.
- 초기 수요 확보는 프랜차이즈 본사 1곳 단위의 다점포 파일럿이 가장 적합하다고 봤다.

`2회차 수정안`
- 핵심 사용자 플로우를 `점포 정보 입력 -> 권장 근무표 생성 -> 현장 변경 요청 수집 -> 리스크 경고 -> 월말 정산용 내보내기`로 고정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `스케줄 앱`이 아니라 `본사형 노동 리스크 가드레일 OS`로 재정의했다.
- 새 가설은 “고객이 가장 아파하는 것은 작성 시간보다도 매장별 룰 일탈과 사후 분쟁 리스크”였다.
- 달라진 점:
  - 점장 편의보다 본사 통제력을 앞세웠다.
  - 주휴/연장 이슈의 사전 경고를 제품 핵심 가치로 올렸다.
  - KPI를 작성 시간에서 인건비율 개선, 초과근로 감소로 바꿨다.
- 난이도 변화:
  - 법률 자문처럼 보이는 기능은 배제해 범위가 정리됐다.
  - 본사별 룰 엔진 설계는 약간 더 복잡해졌다.

`3회차 검증`
- 본사용 운영툴 예산으로 판매할 수 있다면 의사결정은 빨라질 수 있다는 점이 확인됐다.
- 단, POS/근태 연동이 느릴 경우에는 수기 입력만으로도 첫 가치가 나와야 한다는 조건이 붙었다.

`3회차 보강`
- 최종 차별화는 `본사 표준 근무 룰`, `매장별 위험 시간대 경고`, `점장 요청과 본사 승인 흐름 분리`로 정리됐다.

`3회차 수정안`
- 최종 MVP:
  - 본사용 룰 템플릿
  - 점포별 시프트 작성/수정
  - 주휴/연장 위험 경고
  - 인건비율 비교 대시보드
  - 월말 정산 내보내기
- 최종 평가: `강함`
- 이유: 매주 반복되고, 비용 절감 포인트가 뚜렷하며, 본사가 돈을 낼 명분이 있다.

### 4. RecallOS for Dental / Aesthetic Clinics

`1회차 원안`
- 치과와 피부과 등 시술 중심 의원의 재내원 유도, 상담 후 미결제 전환, 노쇼 대응이 인력 의존적이다.

`1회차 검증`
- CRM형 병원 솔루션과 예약 앱은 있지만, 리콜과 예약 회수는 여전히 인력 의존이 컸다. 수요는 `강함`.

`1회차 보강`
- 병원 CRM 전체가 아니라 `리콜과 예약 회수` 문제에 집중해야 한다는 결론이 나왔다.

`1회차 수정안`
- 예약 리마인드, 미방문 환자 재접촉 큐, 상담 후 미결제 분류, 리콜 템플릿을 MVP로 설정했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `치과 교정/임플란트, 피부과 패키지 시술`처럼 객단가가 높은 분야로 좁혔다.
- 새 가설은 “고객은 CRM보다도 노쇼 손실 회수와 상담 후 전환 회수에 더 빨리 반응한다”였다.
- 달라진 점:
  - 범용 병원군을 버리고 고단가 시술군으로 좁혔다.
  - 환자 커뮤니케이션보다 직원 액션 큐를 앞세웠다.
  - 리콜 자동화보다 회수 우선순위 엔진에 더 집중했다.
- 난이도 변화:
  - EMR 깊은 연동을 줄여 난도는 낮아졌다.
  - 개인정보/광고 규제 가이드는 여전히 중요했다.

`2회차 검증`
- Fresha, Boulevard, Phorest 같은 예약/리텐션 시장이 이미 커 니즈는 확실했다.
- 다만 병원별 정책과 법적 표현 제한을 잘라내지 않으면 도입이 느릴 수 있었다.

`2회차 보강`
- 차별화는 `상담 후 미결제 환자 회수`, `내원 단계별 리콜 시나리오`, `직원별 전환 성과 추적`으로 정리됐다.
- 초기 수요층은 마케팅 대행사나 병원 컨설턴트 채널을 통해 묶는 것이 유효하다고 봤다.

`2회차 수정안`
- 핵심 플로우를 `예약/상담 이벤트 유입 -> 회수 우선순위 분류 -> 메시지/콜 태스크 생성 -> 결과 추적`으로 재정의했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `병원 CRM`이 아니라 `노쇼와 미결제 회수 매출 엔진`으로 다시 포지셔닝했다.
- 새 가설은 “병원은 관계관리 도구보다 한 달 내 회수 매출이 늘어나는 도구에 지갑을 연다”였다.
- 달라진 점:
  - 대시보드보다 액션 큐를 핵심으로 올렸다.
  - 재방문 리마인드보다 빈 슬롯 재판매 흐름을 강화했다.
  - 일반 병원보다 패키지 매출 구조가 있는 의원에 집중했다.
- 난이도 변화:
  - EMR 대체를 포기해 범위가 줄었다.
  - 고객별 메시지 시퀀스 템플릿 설계는 조금 더 중요해졌다.

`3회차 검증`
- 수요는 유지됐지만 의료/개인정보 문맥 때문에 세일즈와 도입이 다른 후보보다 조금 무거웠다.
- 고단가 시술군으로 좁히면 경제성은 괜찮지만 시장 크기는 다소 제한될 수 있었다.

`3회차 보강`
- 최종 차별화는 `미결제 환자 회수`, `노쇼 빈자리 재판매`, `직원별 회수 성과 측정`으로 수렴했다.

`3회차 수정안`
- 최종 MVP:
  - 예약/상담 이벤트 유입
  - 회수 우선순위 큐
  - 자동 리콜 시퀀스
  - 취소 대기열/빈자리 회수
  - 담당자 성과 추적
- 최종 평가: `중상`
- 이유: 돈이 새는 문제는 분명하지만 규제와 도입 장벽이 위 후보보다 조금 크다.

### 5. Supplier CAPA Hub

`1회차 원안`

`1회차 검증`

`1회차 보강`
- 전사 QMS 대체가 아니라 `공급사 대응 워크플로우 레이어`로 들어가야 한다는 방향이 나왔다.

`1회차 수정안`

`2회차 원안`
- 1회차 수정안을 바탕으로 `자동차/전자 중견 제조사 품질팀`으로 더 좁혔다.
- 새 가설은 “고객은 품질 관리 전반보다 감사 대응과 협력사 커뮤니케이션 속도 개선에 먼저 돈을 낸다”였다.
- 달라진 점:
  - 모든 제조 업종에서 자동차/전자 중심으로 축소했다.
  - 내부 품질 분석보다 외부 공급사 협업 흐름을 강화했다.
  - 복잡한 승인 체계보다 마감/증빙 회수에 집중했다.
- 난이도 변화:
  - PLM/MES 깊은 연동을 빼서 난도는 낮아졌다.
  - 공급사별 포털 UX와 증빙 구조화 중요도는 올라갔다.

`2회차 검증`
- CAPA 시장은 이미 존재해 수요는 확인됐지만, 세일즈 사이클과 도입 교육이 느리다는 약점은 여전했다.
- 반복 사용성은 높지만 의사결정자가 현장과 멀 수 있다는 리스크가 있었다.

`2회차 보강`
- 차별화는 `이메일/첨부 기반 이슈 카드 자동 생성`, `공급사별 재발 패턴`, `감사 대응 패키지 자동화`로 정리됐다.

`2회차 수정안`
- 핵심 플로우를 `불량 접수 -> 공급사 액션 요청 -> 증빙 회수 -> 내부 승인 -> 재발 패턴 리포트`로 확정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `품질 시스템`보다 `공급사 corrective action 속도 개선 툴`로 다시 정의했다.
- 새 가설은 “속도와 증빙 가시화가 감사 대응보다 더 큰 구매 동기일 수 있다”였다.
- 달라진 점:
  - 분석 툴보다 공급사 응답 추적을 앞세웠다.
  - 공급사별 SLA 개념을 넣었다.
  - 범용 품질팀보다 협력사 이슈가 잦은 조직에 집중했다.
- 난이도 변화:
  - 범위는 더 명확해졌지만, 공급사 포털 온보딩 설계는 어려워졌다.

`3회차 검증`
- 니즈는 분명하지만 다른 후보 대비 세일즈 속도가 느리고 도입 저항이 높았다.
- 큰 계약으로 갈 수 있으나 초기 스타트업이 빠르게 검증하기에는 상대적으로 무거웠다.

`3회차 보강`
- 최종 차별화는 `공급사 응답 속도`, `증빙 누락 방지`, `재발 패턴 가시화`로 정리됐다.

`3회차 수정안`
- 최종 MVP:
  - 이슈 카드 생성
  - 공급사 액션/마감 추적
  - 증빙 수집
  - 감사 대응 패키지
  - 재발 분석
- 최종 평가: `중상`
- 이유: 객단가는 높지만 영업 속도가 느려 초기 사업성 검증은 무겁다.

### 6. GlobalWorker Desk

`1회차 원안`
- 외국인 근로자의 채용 후 서류, 온보딩, 근태 안내, 번역, 생활 지원이 파편화돼 있다.

`1회차 검증`
- HRIS, 비자 대행, 통번역, 노무사 대체재가 있지만 한국 현장 특화 운영툴은 비어 있었다.
- 상주 외국인과 외국인 고용률을 볼 때 수요 강도는 `매우 강함`이었다.

`1회차 보강`
- 비자 자동판정처럼 고위험 기능을 빼고 `운영 자동화`에 집중해야 한다는 결론이 났다.

`1회차 수정안`
- 온보딩 체크리스트, 다국어 공지/FAQ, 서류 만료 알림, 안전교육 확인, 문의 티켓을 MVP로 설정했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `E-9 비중이 높은 제조/물류 사업장`에만 집중했다.
- 새 가설은 “인사팀의 가장 큰 고통은 법률 해석보다도 입사 초기 혼선과 현장 소통 비용”이라는 것이었다.
- 달라진 점:
  - 외국인 근로자 전체에서 E-9 중심으로 좁혔다.
  - 생활지원보다 온보딩/안전교육/공지 전달을 우선했다.
  - 관리자 콘솔과 근로자 앱을 명확히 분리했다.
- 난이도 변화:
  - 고위험 법률 기능을 줄여 난도는 낮아졌다.
  - 다국어 FAQ, 알림 템플릿, 현장 관리자 UX는 중요도가 올라갔다.

`2회차 검증`
- 실제 현장 pain은 매우 명확했고 한국 특화가 강하게 먹힐 수 있었다.
- 경쟁사보다 중요한 것은 완성도 높은 법률 기능이 아니라 “현장 관리자도 바로 쓰는 쉬운 운영 툴”이었다.

`2회차 보강`
- 차별화 포인트는 `언어별 온보딩 패키지`, `서류/재계약 만료 알림`, `안전교육/공지 수신 확인`, `문의 티켓`으로 정리됐다.
- 초기 수요층은 외국인 30명 이상 고용 사업장으로 더 좁혔다.

`2회차 수정안`
- 플로우를 `채용 확정 -> 언어별 온보딩 패키지 발송 -> 서류/교육 완료 추적 -> 현장 문의 대응 -> 재계약/만료 알림`으로 고정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품 정체성을 `외국인 근로자 지원 앱`이 아니라 `현장 관리자용 외국인 인력 운영 OS`로 다시 잡았다.
- 새 가설은 “돈을 내는 사람은 근로자가 아니라 현장 관리자이므로, 관리자 업무량 감소가 제일 큰 가치”였다.
- 달라진 점:
  - 근로자 편의보다 관리자 통제/가시성을 앞세웠다.
  - 생활 정보보다 입사 초기 이탈과 서류 누락 방지에 집중했다.
  - 국적 확장은 후순위로 두고 베트남어/네팔어 등 우선 언어팩 전략을 세웠다.
- 난이도 변화:
  - 범위를 줄여 제품화는 쉬워졌다.
  - 현장별 언어/문화별 템플릿 정교화는 중요해졌다.

`3회차 검증`
- 한국 시장 특화가 강하게 먹히고, 대체재가 분산돼 있어 초기 공략이 가능하다고 판단했다.
- 정책 변경 리스크는 여전하지만 운영 툴 중심으로 자르면 감당 가능한 수준이었다.

`3회차 보강`
- 최종 차별화는 `관리자 콘솔 중심 설계`, `언어별 표준 온보딩`, `서류/재계약/안전교육 추적`, `현장 문의 티켓화`로 수렴했다.

`3회차 수정안`
- 최종 MVP:
  - 관리자 콘솔
  - 근로자 언어별 온보딩 패키지
  - 서류/재계약 만료 알림
  - 안전교육 확인
  - 문의 티켓/FAQ
- 최종 평가: `매우 강함`
- 이유: 한국 시장 특화 가치가 크고, 반복 사용성과 지불 논리가 매우 선명하다.

### 7. PostShip Recovery

`1회차 원안`
- 이커머스 셀러는 배송 지연, 주소 오류, 수취 거부, 반품 사유 회수에서 매출과 리뷰를 동시에 잃는다.

`1회차 검증`
- AfterShip, Narvar, Loop 등 글로벌 시장이 이미 형성돼 있어 니즈는 확실했다.
- 한국 셀러는 국내 택배사와 알림톡 중심 워크플로우가 중요했다.

`1회차 보강`
- 일반 배송추적이 아니라 `예외 주문 회수`와 `반품 인사이트`에 집중해야 한다는 방향이 나왔다.

`1회차 수정안`
- 배송 예외 감지, 고객 연락 자동화, 주소 수정 회수, 반품 사유 구조화가 MVP로 정리됐다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `월 1천~2만 건 주문의 D2C 뷰티/패션 브랜드`에 더 좁혔다.
- 새 가설은 “고객이 돈을 내는 이유는 추적 페이지가 아니라 예외 주문에서 잃는 매출 회수”라는 것이었다.
- 달라진 점:
  - 추적 경험 개선을 거의 버렸다.
  - 주소 오류/장기 미배송/수취 거부 등 예외 유형별 회수 흐름을 강화했다.
  - CS팀 업무 대신 매출 회수 KPI를 전면에 세웠다.
- 난이도 변화:
  - 브랜딩 기능을 빼서 난도는 낮아졌다.
  - 택배사/몰 연동 중요도는 여전히 높았다.

`2회차 검증`
- Narvar, Loop 등 기존 시장은 강했지만 “한국 택배 + 알림톡 + 예외 회수”는 분명한 로컬 틈이 있었다.
- 반복 사용성과 ROI는 매우 좋게 유지됐다.

`2회차 보강`
- 차별화는 `예외 주문 자동 분류`, `고객별 회수 액션 추천`, `반품 사유를 상품/CS 인사이트로 연결`하는 구조로 정리됐다.

`2회차 수정안`
- 플로우를 `주문/운송장 유입 -> 예외 감지 -> 고객/CS 액션 생성 -> 해결 -> 재발 원인 리포트`로 고정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `포스트퍼체이스 도구`보다 `예외 주문 회수 매출 OS`로 다시 규정했다.
- 새 가설은 “셀러는 알림보다도 환불/반품으로 새는 돈을 줄이는 자동화에 반응한다”였다.
- 달라진 점:
  - 고객 만족보다 회수 금액과 처리 시간으로 KPI를 바꿨다.
  - 반품 사유를 제품 개선 인사이트로 이어붙였다.
  - 타겟을 재고/반품 부담이 큰 카테고리로 더 좁혔다.
- 난이도 변화:
  - 범위는 더 날카로워졌고, 통합 화면 설계는 단순해졌다.
  - 사유 분류 정확도와 연동 품질은 중요도가 높아졌다.

`3회차 검증`
- 강한 경쟁군이 있어도 한국형 예외 회수 워크플로우는 여전히 유망했다.
- 주문량이 어느 정도 있는 셀러에게는 구독료보다 회수 매출 논리가 더 쉽게 먹힐 것으로 봤다.

`3회차 보강`
- 최종 차별화는 `한국 택배 예외 회수`, `알림톡 기반 고객 액션`, `반품 사유 데이터화`로 수렴했다.

`3회차 수정안`
- 최종 MVP:
  - 예외 주문 감지
  - 회수 액션 큐
  - 주소 수정/재배송 플로우
  - 반품 사유 구조화
  - 예외 KPI 대시보드
- 최종 평가: `강함`
- 이유: 일간 사용성이 높고, ROI가 직접적이며, 로컬 특화 여지가 충분하다.

### 8. InterviewFlow

`1회차 원안`
- 고볼륨 채용에서 면접 평가 기준이 흐리고 피드백 누락이 잦다.

`1회차 검증`
- ATS 시장이 성숙해 니즈는 분명하지만 경쟁도 강했다. 수요 강도는 `중간`.

`1회차 보강`
- 범용 ATS가 아니라 `현장 면접 실행 레이어`로 들어가야 한다는 방향이 나왔다.

`1회차 수정안`
- 모바일 면접평가, 직무 템플릿, 회수 리마인드, 합격 기준 대시보드 중심으로 수정했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `1년에 300명 이상 현장직을 뽑는 외식/리테일/물류 기업`으로 좁혔다.
- 새 가설은 “고객이 원하는 것은 ATS 교체가 아니라 면접관 평가 회수율과 품질 통제”였다.
- 달라진 점:
  - 채용 전 과정보다 면접 입력 순간에 집중했다.
  - HR용 관리보다 현장 관리자용 1분 평가 UX를 앞세웠다.
  - ATS 통합은 보조 역할로 밀어냈다.
- 난이도 변화:
  - 제품 범위는 줄어들어 난도는 낮아졌다.
  - 조직 도입과 교육 설계 중요도는 더 올라갔다.

`2회차 검증`
- Greenhouse, Lever, Workable 등 대체재가 강해서 독립 제품으로 큰 시장을 먹기 어렵다는 점은 여전했다.
- 다만 ATS가 약한 현장직 대량채용 기업에는 틈이 있었다.

`2회차 보강`
- 차별화는 `모바일 초고속 평가`, `직무별 합격 신호 구조화`, `면접관 회수율/품질 분석`으로 정리됐다.

`2회차 수정안`
- 플로우를 `면접관 배정 -> 모바일 평가 입력 -> HR 리뷰 -> 합격/보류 -> 면접관 품질 피드백`으로 단순화했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `면접 협업 툴`이 아니라 `현장 면접 준수율 관리 툴`로 재정의했다.
- 새 가설은 “문제는 판단의 질보다도 기록 누락과 기준 불일치에 있다”였다.
- 달라진 점:
  - 후보자 경험보다 내부 운영 품질을 앞세웠다.
  - 채용품질 예측보다 평가 회수율과 표준화 준수율을 KPI로 삼았다.
  - ATS 없는 고객군을 우선 타겟으로 정리했다.
- 난이도 변화:
  - 예측 AI 비중이 줄어 난도는 낮아졌다.
  - 도입 시 교육/변화관리 의존성은 여전히 높았다.

`3회차 검증`
- 반복 사용성은 있으나 범용 툴에 흡수될 위험은 끝까지 남았다.
- 초기 사업으로는 가능하지만 최상위 우선순위로 보기엔 방어력이 약했다.

`3회차 보강`
- 최종 차별화는 `1분 평가 UX`, `회수율 관리`, `직무별 표준 기준` 정도로 압축됐다.

`3회차 수정안`
- 최종 MVP:
  - 모바일 평가 입력
  - 직무 템플릿
  - 회수 리마인드
  - 면접관 준수율 대시보드
  - ATS 메모 동기화
- 최종 평가: `중간`
- 이유: pain은 있지만 독립 시장 방어력은 상대적으로 약하다.

### 9. SalesFollow OS

`1회차 원안`
- B2B 영업 미팅 후 제안서, 샘플, 견적, 다음 액션이 담당자 개인 역량에 묶인다.

`1회차 검증`
- Gong, Fireflies, CRM 등이 이미 강해 수요는 강하지만 경쟁은 심했다.

`1회차 보강`
- 일반 회의요약이 아니라 `복잡한 후속처리 자동화`로 차별화해야 한다는 방향이 나왔다.

`1회차 수정안`
- 미팅 기록 요약, 후속 태스크 자동 생성, 견적/샘플/문서 체크리스트를 MVP로 정리했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `산업재/수출 영업팀`처럼 후속 문서가 많은 조직에 더 집중했다.
- 새 가설은 “영업팀이 진짜 잃는 것은 회의록 부재가 아니라 후속 문서/샘플 누락으로 인한 딜 소실”이었다.
- 달라진 점:
  - 범용 세일즈 팀을 버리고 산업재 쪽으로 좁혔다.
  - 회의 요약보다 후속 액션 completeness를 전면에 세웠다.
  - CRM 대체 포지셔닝을 내려놨다.
- 난이도 변화:
  - 녹취 AI 비중을 줄여 기술 난도는 낮아졌다.
  - 업종별 후속 템플릿 설계 중요도는 높아졌다.

`2회차 검증`
- 경쟁이 매우 강하고, 사용자 교육 없이도 자연스럽게 자리잡기 어렵다는 리스크가 컸다.
- 특정 업종으로 좁히면 가능하지만 여전히 강한 상위 툴과 부딪혔다.

`2회차 보강`
- 차별화는 `견적/샘플/문서 체크리스트`, `후속 지연 알림`, `매니저용 후속 누락 리포트`로 정리됐다.

`2회차 수정안`
- 플로우를 `미팅 메모 입력 -> 후속 패키지 생성 -> 담당자 할당 -> 완료 추적 -> 파이프라인 리스크 표시`로 고정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `회의 도구`보다 `후속 누락 방지 엔진`으로 다시 정의했다.
- 새 가설은 “산업재 영업에서 후속 문서/샘플 완료율이 실제 매출과 더 강하게 연결된다”였다.
- 달라진 점:
  - 음성 회의 AI 비중을 더 낮췄다.
  - 문서와 샘플 흐름을 더 중심에 놓았다.
  - 영업관리자용 리스크 화면을 강화했다.
- 난이도 변화:
  - 제품 구현은 명확해졌지만, 시장에서의 차별화는 여전히 쉽지 않았다.

`3회차 검증`
- 니즈는 있지만 레드오션 압박이 컸고, 초기 스타트업이 들어가기엔 설득 비용이 크다는 판단이 유지됐다.

`3회차 보강`
- 최종 차별화는 `업종별 후속 템플릿`, `체크리스트 completeness`, `매니저용 누락 리포트` 정도로 압축됐다.

`3회차 수정안`
- 최종 MVP:
  - 미팅 후속 패키지 생성
  - 문서/샘플 체크리스트
  - 완료 추적
  - 리스크 알림
  - CRM 메모 동기화
- 최종 평가: `중간`
- 이유: pain은 명확하지만 경쟁 회피 가능성이 낮다.

### 10. Resident MaintOps

`1회차 원안`
- 아파트/오피스텔/상가의 시설 민원, 협력업체 배정, 공지, 비용 승인 흐름이 분절돼 있다.

`1회차 검증`
- 관리 앱과 시설관리 솔루션이 이미 존재해 수요는 분명했으나 의사결정이 느릴 수 있었다.

`1회차 보강`
- 커뮤니티 앱 전체가 아니라 `유지보수 운영 워크플로우`에 집중해야 한다는 방향이 나왔다.

`1회차 수정안`
- 민원 접수, 작업지시, 업체 배정, 사진 증빙, 공지, 설비 이력을 MVP로 설정했다.

`2회차 원안`
- 1회차 수정안을 바탕으로 `300~1,500세대 단지의 위탁관리사/관리사무소`에 집중했다.
- 새 가설은 “입주민 경험 자체보다 관리사무소 직원 업무량과 협력업체 추적 문제를 해결해야 돈을 낸다”였다.
- 달라진 점:
  - 입주민 앱보다 관리사무소 툴을 앞세웠다.
  - 커뮤니티 기능을 거의 다 제거했다.
  - 누수/승강기/주차 같은 자주 터지는 민원 카테고리 중심으로 좁혔다.
- 난이도 변화:
  - 커뮤니티/소셜 요소를 빼서 난도는 낮아졌다.
  - 협력업체 배정/증빙 UX 중요도는 높아졌다.

`2회차 검증`
- 아파트너류 사례가 있는 만큼 수요는 확실했지만, 도입 결정과 계약 구조가 느릴 가능성이 컸다.
- 위탁관리사 본부 단위로 잡으면 판매 효율이 나아질 수 있다는 점이 확인됐다.

`2회차 보강`
- 차별화는 `민원 자동 분류`, `작업지시와 업체 배정 연결`, `반복 설비 이력`, `공지 도달률 관리`로 정리됐다.

`2회차 수정안`
- 플로우를 `민원 접수 -> 우선순위 분류 -> 업체/직원 배정 -> 완료 증빙 -> 입주민 안내`로 고정했다.

`3회차 원안`
- 2회차 수정안을 바탕으로 제품을 `주거 플랫폼`이 아니라 `관리사무소 유지보수 오퍼레이션 OS`로 다시 정의했다.
- 새 가설은 “구매를 움직이는 것은 입주민 만족도 자체보다도 전화 민원 감소와 협력업체 관리 효율”이었다.
- 달라진 점:
  - 입주민용 기능을 더 줄였다.
  - 관리사무소와 협력업체 간 흐름을 핵심으로 올렸다.
  - KPI를 앱 활성도에서 평균 처리시간/재민원율 감소로 바꿨다.
- 난이도 변화:
  - 제품 범위는 줄었지만, B2B2C 영업 구조의 느림은 남아 있었다.

`3회차 검증`
- 니즈는 확실하지만 판매 속도와 의사결정 구조 때문에 초기 스타트업 우선순위는 조금 떨어졌다.

`3회차 보강`
- 최종 차별화는 `민원-작업지시-업체배정 일원화`, `설비 유형별 이력`, `사진 증빙과 공지 연결`로 수렴했다.

`3회차 수정안`
- 최종 MVP:
  - 민원 접수/분류
  - 작업지시
  - 업체 배정
  - 전후 사진 증빙
  - 설비 이력/공지
- 최종 평가: `중상`
- 이유: 반복성은 높지만 영업 구조가 느려 초반 속도는 다소 제한될 수 있다.

## 왜 이 순위인가

- 1~5위는 3회 루프를 돌린 뒤에도 `시장성`, `반복 사용성`, `지불 논리`, `한국 현장 특화 여지`, `파일럿 속도`가 동시에 유지됐다.
- 6~8위는 분명히 돈이 새는 문제를 해결하지만 규제, 판매 사이클, 의사결정 구조 때문에 초기 속도가 조금 느리다.
- 9~10위는 pain은 있으나 이미 강한 범용 툴과 겹치는 영역이 많아 초기 차별화 비용이 높다.

## 공통 패턴

- 살아남은 아이디어는 전부 `올인원 플랫폼`에서 `반복되는 단일 워크플로우 개선`으로 진화했다.
- 세일즈 메시지는 기능 설명보다 `한 달 안에 줄어드는 손실`로 설계하는 것이 더 강했다.
- 1회차에서 넓었던 타겟은 2회차에서 업종/규모/역할 기준으로 좁아졌고, 3회차에서는 `돈 내는 의사결정자` 기준으로 다시 정리됐다.
