가정:
- 기존 3~10회차 로그를 기준으로 에이전트 성능을 평가했다.
- 신규 5개 아이디어는 기존 상위 10개와 겹치지 않는 vertical workflow만 선정했다.
- 이번 신규 실험은 `개선된 프롬프트 기준 1회 완결 루프`로 정리했다.

## 1. 에이전트 평가

### 1번 에이전트 평가

- 종합 점수: `8.7/10`
- 잘한 점:
  - 회차가 갈수록 범위를 잘 줄였다.
  - `누가 돈을 내는지`, `어디서 손실이 나는지`, `MVP를 어디까지 자를지`를 후반부에 잘 잡았다.
  - 상위권 아이디어들은 거의 모두 `올인원 -> 단일 워크플로우`로 수렴시켰다.
- 아쉬운 점:
  - 중간 회차에서 대시보드나 보조 기능이 다시 커지는 구간이 있었다.
  - 몇몇 아이디어는 `실사용자`와 `결제자`의 분리가 늦게 명확해졌다.
  - 초반부터 가격 가설과 파일럿 범위를 더 강하게 고정했으면 더 빨리 좁혀졌을 것이다.
- 개선 포인트:
  - 첫 문단에 `한 문장 pain + 결제자 + 반복 빈도`를 강제한다.
  - 매 회차마다 `버린 것`을 명시하게 한다.
  - `6주 파일럿`을 넘는 범위를 초반부터 잘라낸다.

### 2번 에이전트 평가

- 종합 점수: `8.3/10`
- 잘한 점:
  - 유사 서비스와 대체재를 꾸준히 찾아서 카테고리 존재 여부를 확인했다.
  - 아이디어가 살아남는 이유뿐 아니라 `왜 안 살 수도 있는지`를 꽤 잘 드러냈다.
  - 후반부에는 `좁힐수록 설득력이 높아진다`는 신호를 일관되게 줬다.
- 아쉬운 점:
  - 일부 검증은 정성 코멘트가 강하고, `폐기/보류` 판정이 더 과감할 수 있었다.
  - 출처 강도와 데이터 신선도를 회차별로 더 선명하게 구분하지는 못했다.
  - `대체재가 있음 -> 그래서 설 자리가 없음/있음`의 논리가 더 날카로울 수 있었다.
- 개선 포인트:
  - 각 아이디어마다 `살림 / 조건부 살림 / 보류 / 폐기` 판정을 강제한다.
  - `돈을 안 낼 이유`를 반드시 별도 항목으로 둔다.
  - 링크와 카테고리 근거를 짧게라도 남기게 한다.

### 3번 에이전트 평가

- 종합 점수: `8.5/10`
- 잘한 점:
  - 차별화 포인트를 `예쁜 문구`가 아니라 `팔 수 있는 채널`, `실험 KPI`, `수익화`와 연결하려고 했다.
  - 1번 에이전트가 다시 좁힐 수 있게 좋은 질문을 남겼다.
  - 특히 상위권 아이디어에서 `첫 고객 채널`과 `성과형 메시지`를 선명하게 만들었다.
- 아쉬운 점:
  - 일부 회차에서는 GTM이 다소 넓고 일반적이었다.
  - 채널을 많이 제시하는 대신 `첫 번째로 어디를 찌를지`를 더 강하게 골라줄 수 있었다.
  - `포기해야 하는 경우`를 더 명확히 끊지 못한 구간이 있었다.
- 개선 포인트:
  - 각 아이디어마다 `첫 고객 확보 채널 1개`를 우선 지정한다.
  - `6주 실험 KPI`를 수치형으로 강제한다.
  - 못 살리는 아이디어는 왜 못 사는지 분명히 적게 한다.

## 2. 개선된 프롬프트

### 개선된 1번 에이전트 프롬프트

```text
당신은 개선된 1번 에이전트입니다. 역할은 강한 vertical wedge를 가진 아이디어를 발굴하고, 피드백을 받아 더 좁고 더 잘 팔리는 방향으로 재정의하는 것입니다.

규칙:
- 범용 툴, 올인원 플랫폼, 취미성 앱은 금지입니다.
- 각 아이디어 첫 문단에서 반드시 `누가`, `언제 반복적으로`, `왜 돈을 내는지`, `어떤 한 문장 pain인지`를 드러내세요.
- 매 회차마다 반드시 `지킨 것`과 `버린 것`을 분리하세요.
- 실사용자와 결제자를 구분하세요.
- 6주 파일럿으로 검증 가능한 MVP만 남기세요.
- 기존 아이디어와 겹치는 카테고리는 피하세요.

출력 형식:
- 아이디어명
- 한 문장 pain
- 구체 타겟 조직/직무
- 반복 사용 이유
- 돈을 낼 이유
- 지킨 것
- 버린 것
- 6주 파일럿형 MVP
```

```text
당신은 개선된 2번 에이전트입니다. 역할은 시장 검증과 리스크 평가입니다.

규칙:
- 각 아이디어마다 반드시 웹 검색으로 유사 서비스/대체재를 찾으세요.

- 치명적 리스크는 3개까지 적고, 링크나 출처를 최대한 남기세요.

- 생존 판정
- 3번 에이전트로 넘길 메모
```

## 3. 신규 5개 아이디어: 개선된 1번 에이전트 초안

### 1. ServiceDispatch Proof

- 한 문장 pain: 상업용 주방/카페 장비 유통사의 A/S 코디네이터는 고장 접수, 기사 배정, 부품 필요 여부, 보증/유상 판정, 완료 증빙을 전화와 카카오톡으로 처리하다가 SLA를 놓치고 청구도 빼먹는다.
- 아주 구체적인 타겟 조직/직무: `상업용 주방 장비, 커피머신, 냉장·냉동 장비`를 공급하는 10~80명 규모 유통사의 서비스 운영팀장과 디스패처
- 반복 사용 이유: 서비스 접수, 기사 배정, 사진 회수, 부품 재방문은 거의 매일 반복된다.
- 돈을 낼 이유: SLA 위반, 무상/유상 청구 누락, 재방문 증가가 바로 마진 손실로 연결된다.
- 이번 아이디어에서 의도적으로 버린 범위: 예방정비, 재고 ERP, 회계, 풀 FSM 전반
- 6주 파일럿형 MVP:
  - 고장 접수 인박스
  - 기사 배정 보드
  - 무상/유상 체크리스트
  - 사진/서명 완료 증빙
  - 미청구 작업 큐

### 2. CareFamily Log

- 한 문장 pain: 요양원과 주간보호센터는 사고, 상태 변화, 보호자 동의, 병원 외출 기록을 전화와 수기로 관리해 민원과 추적 누락이 반복된다.
- 아주 구체적인 타겟 조직/직무: 입소자 50~200명 규모 요양원·주간보호센터의 시설장, 사회복지사, 보호자 커뮤니케이션 담당
- 반복 사용 이유: 보호자 공지, 사고 기록, 상태 전달, 동의 회수는 거의 매일 발생한다.
- 돈을 낼 이유: 민원, 분쟁, 감독 대응, 직원 시간 낭비를 동시에 줄일 수 있다.
- 이번 아이디어에서 의도적으로 버린 범위: EMR, 임상 의사결정, 처방, 보험청구
- 6주 파일럿형 MVP:
  - 사고/상태 변화 기록 템플릿
  - 보호자 공지/동의 요청
  - 확인 로그
  - 담당자 후속 큐
  - 사건별 타임라인

### 3. ChangeOrder Cashflow OS

- 한 문장 pain: 상가·오피스 인테리어 시공사는 추가 공사와 변경 요청이 구두 승인으로 흘러가다가 돈은 못 받는데 공정은 이미 진행된다.
- 아주 구체적인 타겟 조직/직무: 5~50명 규모 상업 인테리어 시공사의 PM, 현장소장, 대표
- 반복 사용 이유: 변경 요청, 단가 조정, 추가 자재, 진행 청구는 프로젝트마다 계속 반복된다.
- 돈을 낼 이유: 승인 누락과 청구 지연은 바로 현금흐름 악화로 이어진다.
- 이번 아이디어에서 의도적으로 버린 범위: 전체 공정관리, BIM, 자재발주, 원가회계
- 6주 파일럿형 MVP:
  - 현장 변경 요청 등록
  - 모바일 승인 요청
  - 추가 금액/일정 영향 기록
  - 진행 청구 연결
  - 미승인 변경 작업 큐

### 4. Backorder Substitute Desk

- 한 문장 pain: 식자재 유통사는 품절, 대체품 제안, 미납품, 크레딧 처리를 전화와 엑셀로 때우다가 마진과 고객 신뢰를 같이 잃는다.
- 아주 구체적인 타겟 조직/직무: 지역 식자재·푸드서비스 유통사의 주문데스크 매니저, CS 리드, 영업지원 팀장
- 반복 사용 이유: 품절과 대체품 협의는 주문량이 많을수록 매일 발생한다.
- 돈을 낼 이유: 잘못된 대체·미납품 대응은 바로 취소, 할인, CS 인건비 증가로 이어진다.
- 이번 아이디어에서 의도적으로 버린 범위: 전체 ERP, 창고관리, 회계, 차량배차
- 6주 파일럿형 MVP:
  - 품절/백오더 큐
  - 대체품 제안 템플릿
  - 고객 승인 로그
  - 미납품/크레딧 추적
  - 마진 영향 리포트

### 5. FleetClaim Ops

- 한 문장 pain: 배송·서비스 차량을 많이 운영하는 회사는 사고 후 사진, 운전자 진술, 보험사 서류, 수리 승인, 대차 조정을 흩어진 채널로 처리하다가 차량 가동률이 떨어진다.
- 아주 구체적인 타겟 조직/직무: 50~500대 규모의 라스트마일 배송사, 필드서비스 운영사, 렌터카 운영사의 차량 운영팀장과 사고 담당 매니저
- 반복 사용 이유: 사고 자체는 상시가 아니어도 일정 규모 이상이면 매주 반복되고 처리 기간이 길다.
- 돈을 낼 이유: 사고 차량 한 대당 가동 중단 비용과 클레임 누락 비용이 크다.
- 이번 아이디어에서 의도적으로 버린 범위: 텔레매틱스, 일반 정비관리, 차량 위치추적, 보험 코어 시스템
- 6주 파일럿형 MVP:
  - 사고 접수 모바일 폼
  - 사진/진술 수집
  - 보험/정비소 진행상태 보드
  - 대차 필요 여부 체크
  - 사고당 소요일/누락 문서 리포트

## 4. 신규 5개 아이디어: 개선된 2번 에이전트 검증

### 1. ServiceDispatch Proof

- 유사 서비스/대체재:
  - ServiceTrade: https://servicetrade.com/features/dispatch/
  - Equipment Pack: https://www.equipmentpack.com/
  - FieldBin commercial kitchen repair: https://www.fieldbin.com/commercial-kitchen-repair-software/
  - Service Fusion commercial food equipment: https://www.servicefusion.com/commerical-food-equipment-software
- 수요가 강한 근거:
  - 상업용 장비 서비스 업계에는 이미 dispatch/FSM 카테고리가 존재한다.
  - 다만 범용 FSM은 `보증/유상 판정 + 사진 증빙 + 미청구 작업 큐` 같은 서비스 운영 손실 지점에 덜 특화돼 있다.
- 반복 사용 가능성 평가: `높음`
- 돈을 낼 이유 / 안 낼 이유:
  - 낼 이유: SLA 위반 감소, 재방문 감소, 미청구 감소
  - 안 낼 이유: 기존 FSM을 이미 쓰는 조직은 교체 의사가 낮을 수 있다
- 치명적 리스크 3개:
  - 기존 FSM과의 포지션 중복
  - 기사 현장 입력 저항
  - 부품/청구 연동 요구 증가
- 생존 판정: `조건부 살림`
- 3번 에이전트로 넘길 메모: 풀 FSM이 아니라 `보증/유상 판정 + 완료 증빙 + 미청구 회수` 레이어로 더 좁혀야 설 자리가 생긴다.

### 2. CareFamily Log

- 유사 서비스/대체재:

## 5. 신규 5개 아이디어: 개선된 3번 에이전트 방식 보강

### 1. ServiceDispatch Proof

- 유지할 강점: 서비스 누락과 미청구는 바로 마진 손실로 이어진다.
- 버릴 요소: 일반 dispatch, 전체 FSM, 예방정비 확장
- 차별화 포인트:
  - `보증/유상 판정 큐`
  - `사진/서명 완료 증빙`
  - `미청구 작업 회수 큐`
- 첫 고객 확보 채널: 상업용 장비 유통사 서비스팀장 네트워크와 업계 A/S 대행 파트너
- 6주 실험 KPI:
  - 미청구 작업 건수 감소율
  - SLA 초과 작업 감소율
  - 재방문률 감소율
- 수익화 방향: 기사 수 기준 월 구독 + 고급 청구 리포트 업셀
- 살리는 조건 또는 포기 이유: `보증/유상 판정`과 `미청구 회수`를 핵심으로 박으면 살릴 수 있다.
- 1번 에이전트에게 줄 질문:
  - 기사 입력을 최소화하려면 현장 완료 증빙을 어떻게 단순화할 것인가?
  - 기존 FSM을 교체하지 않고도 들어갈 가장 작은 wedge는 무엇인가?

### 2. CareFamily Log

- 유지할 강점: 사고/상태 변화와 보호자 커뮤니케이션은 매우 반복적이다.
- 버릴 요소: 가족 포털, 활동 공유, 커뮤니티 기능
- 차별화 포인트:
  - `사고/상태 변화 공지 템플릿`
  - `동의 요청과 확인 로그`
  - `분쟁 대응용 사건 타임라인`
- 첫 고객 확보 채널: 요양시설 컨설턴트와 운영 법인 본부
- 6주 실험 KPI:
  - 보호자 응답 지연 시간
  - 누락된 공지/동의 건수
  - 사고 후속 처리 시간
- 수익화 방향: 시설당 월 구독
- 살리는 조건 또는 포기 이유: 가족 앱이 아니라 `사건 커뮤니케이션 운영툴`로 매우 좁혀야만 살릴 수 있다.
- 1번 에이전트에게 줄 질문:
  - 어떤 사건 유형부터 시작해야 보호자 동의/공지 pain이 가장 큰가?
  - 민감정보를 최소화한 구조로도 충분한 가치가 나는가?

### 3. ChangeOrder Cashflow OS

- 살리는 조건 또는 포기 이유: full PM이 아니라 `돈 못 받는 변경작업` 한 문제로 파고들면 살릴 수 있다.
- 1번 에이전트에게 줄 질문:

  - `대체 승인 로그`
  - `크레딧/미납품 회수 추적`
- 첫 고객 확보 채널: 지역 식자재 유통사 주문데스크 팀장과 CS 리드

  - 주문 취소율 감소
  - 크레딧 누락 감소

- 1번 에이전트에게 줄 질문:
  - 고객 승인 수단은 문자/카톡/전화 로그 중 무엇으로 표준화할 것인가?

### 5. FleetClaim Ops

- 유지할 강점: 사고 차량 downtime은 돈으로 환산되기 쉽다.

- 1번 에이전트에게 줄 질문:

  - 보험사 직접 연동 없이도 첫 가치가 충분히 나오는가?

## 7. 이번 실험에서 확인된 프롬프트 개선 효과

- 1번 개선 효과:
  - 초반부터 vertical wedge가 선명해졌다.
  - `버린 것`을 함께 적게 하니 범위가 훨씬 덜 흔들렸다.
- 2번 개선 효과:
  - `돈을 안 낼 이유`와 `생존 판정`이 들어가니 아이디어가 더 빨리 좁혀졌다.
  - 조건부 살림과 보류를 나누면서 우선순위가 명확해졌다.
- 3번 개선 효과:
  - GTM이 추상적인 전략 문구보다 `첫 고객 채널`과 `6주 KPI` 중심으로 바뀌었다.
  - `살리는 조건`을 쓰게 하니 1번이 다시 수정할 방향이 선명해졌다.
