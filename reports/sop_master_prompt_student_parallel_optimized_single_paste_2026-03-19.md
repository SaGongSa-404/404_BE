# 학생 팀용 프로젝트 아이디어 발굴 SOP Single Paste Parallel Optimized
작성일: 2026-03-19

## 문서 목적
이 문서는 대학생 팀이 사용할 `프로젝트 아이디어 발굴/판별/보강`용 SOP의 병렬 최적화 버전이다.

목표는 기존 SOP와 거의 같은 판단 효과를 유지하면서도, 에이전트 소모 시간과 중복 작업을 줄이는 것이다.

또한 이 문서는 단순히 `비슷한 서비스가 있는데 좋아 보여서 따라 만든 프로젝트`가 아니라, 아래를 설명할 수 있는 포트폴리오용 프로젝트 아이디어를 찾는 데 초점을 둔다.

- 왜 이 문제를 골랐는가
- 유사 서비스가 실제로 존재하는가
- 수요가 구체적으로 확인되는가
- 그 수요나 불편함을 뒷받침하는 공식 자료가 존재하는가
- 기존 서비스와 비교해 어떤 관찰과 판단으로 이 아이디어를 선택했는가

이 문서가 찾으려는 아이디어 조건은 기존과 같다.

- 총 프로젝트 기간은 약 5개월이다.
- 이 중 2개월 안에는 첫 배포가 가능해야 한다.
- 배포 이후 남은 약 3개월 동안 지표, 설문, 인터뷰를 바탕으로 개선 사이클을 돌릴 수 있어야 한다.

즉, 이 문서는 `5개월짜리 학생 프로젝트에 맞는 아이디어`를 고르는 문서이며, 핵심 차이는 `순차 처리` 대신 `병렬 처리`를 적극 활용한다는 점이다.

## 병렬 최적화 핵심 원칙
1. 순차 의존을 줄인다.
2. 초반에는 넓고 얕게, 후반에는 좁고 깊게 본다.
3. 동일한 검색을 여러 에이전트가 반복하지 않게 한다.
4. 2회차 이후에는 전체 재서술 대신 변화점만 본다.
5. 아이디어는 개별 1개씩이 아니라 `2개 묶음 배치`로 검토한다.
6. 2번과 3번 에이전트는 같은 입력을 받아 동시에 일한다.
7. 취합 에이전트는 긴 요약보다 `다음 컷 기준`과 `생존 판정`만 빠르게 정리한다.

## 기존 대비 가장 큰 구조 변화
- 기존: `1번 -> 2번 -> 3번 -> 1번`
- 변경: `1번 -> (2번과 3번 병렬) -> 취합 -> 1번`

이 구조를 쓰면 2번 에이전트의 현실성 검토와 3번 에이전트의 배포 후 학습 가능성 검토를 동시에 돌릴 수 있다.

## 기본 실행 설정
- 원본 아이디어 수: 8개
- 기본 핵심 루프: 3회
- 최종 선별 목표: 메인 후보 1개 + 예비 후보 1개
- 검토 배치 크기: 2개 아이디어씩
- 기준 문서 소스 오브 트루스: `reports/current/master_report.md`
- 최신 점수표: `reports/current/scorecard.md`
- 공통 리서치 메모: `reports/current/common_research_memo.md`
- 에이전트 로그 폴더: `reports/agent_logs/`
- 후속 비교 문서 폴더: `reports/productization/`

## 이 SOP가 찾으려는 아이디어 조건
- PO 2, FE 3, 디자이너 1, BE 2 기준의 대학생 팀이 현실적으로 만들 수 있는가
- 2개월 안에 첫 배포 가능한 MVP 범위로 자를 수 있는가
- 배포 후 3개월 동안 개선 실험을 할 만큼 사용 데이터가 생길 수 있는가
- 첫 사용자를 학교, 동아리, 커뮤니티, SNS, 지인 네트워크, 로컬 그룹 등에서 확보할 수 있는가
- 설문과 인터뷰로 의미 있는 피드백을 받을 수 있는가
- 유사 서비스 또는 대체재가 실제로 존재하는가
- 유사 서비스가 있어도 차별화 포인트를 만들 수 있는가
- 유사 서비스가 있어도 더 좁은 타겟이나 더 선명한 wedge로 접근할 수 있는가
- 수요를 뒷받침할 구체적 근거가 있는가
- 수요나 불편함을 설명할 공식 자료가 있는가
- 포트폴리오에서 문제 정의, 시장 관찰, 차별화 판단을 설명할 수 있는가
- 너무 제조업형 B2B vertical software나 복잡한 기관/계약 중심 구조가 아닌가

## 빠른 탈락 규칙
아래 중 2개 이상이면 기본적으로 `보류` 검토:

- 유사 서비스가 거의 없는데 그 이유가 시장 부재처럼 보임
- 수요 근거가 추상적이고 구체적인 사용 맥락이 안 보임
- 수요를 설명할 공식 자료를 전혀 찾지 못했고, 비공식 추정만 남음
- 유사 서비스는 분명히 많은데, 차별화 포인트나 타겟 축소 전략이 전혀 안 보임
- 2개월 내 첫 배포 범위로 안 잘림
- 배포 후 3개월 동안 볼 지표가 잘 안 떠오름
- 첫 사용자 채널이 불명확함
- 사용 후 다시 돌아올 상황이 약함
- 경쟁 서비스 대비 왜 써야 하는지 설명이 약함
- 팀 역량 대비 구현/운영 난도가 너무 높음
- 인터뷰나 설문으로 배울 수 있는 포인트가 너무 약함

## 시간 절약용 운영 규칙
### 1. 공통 검색 선행
라운드마다 먼저 `공통 리서치 메모`를 만든다.

이 메모에는 아래만 짧게 정리한다.
- 이번 라운드 후보들이 공통으로 마주치는 경쟁 서비스 유형
- 사용자가 기대하는 기본 기능 수준
- 실제로 수요가 보이는 반복 불만, 질문, 우회 행동
- 수요/불편을 뒷받침하는 공식 자료 유형
- 학생 팀이 접근 가능한 첫 사용자 채널 후보
- 구현/배포 시 공통 리스크

2번, 3번 에이전트는 이 메모를 참고해서 아이디어별 판단만 한다.

### 2. 배치 처리
8개 아이디어를 아래처럼 2개씩 나눈다.

- 배치 A: 아이디어 1~2
- 배치 B: 아이디어 3~4
- 배치 C: 아이디어 5~6
- 배치 D: 아이디어 7~8

2번 에이전트는 각 배치를 병렬 검토하고, 3번 에이전트도 같은 배치를 병렬 검토한다.

### 3. 라운드별 밀도 조절
- 1회차: 8개 전체를 짧게 스크리닝
- 2회차: 상위 4개만 정밀 검토
- 3회차: 상위 2개만 깊게 비교

즉, 모든 후보를 끝까지 같은 밀도로 다루지 않는다.

### 4. 변화점만 재검토
2회차 이후 2번/3번 에이전트는 아래만 다시 본다.

- 지난 회차 대비 더 좁아진 점
- 아직 남은 핵심 리스크
- 판정이 바뀌어야 하는 이유

```text
    agent3_batchC.md

```

파일 규칙:
- 기준 문서는 항상 `reports/current/master_report.md`
- 공통 검색 메모는 항상 `reports/current/common_research_memo.md`
- 회차별 종합본은 `reports/rounds/round0X_master.md`
- 상위 3개 비교 문서는 `reports/productization/top3_comparison.md`
- 최종 1개 추천안은 `reports/productization/top1_project_brief.md`

## 전체 실행 순서
1. 필요한 폴더가 없으면 생성한다.
2. 1번 에이전트를 실행해 원본 아이디어 8개를 만든다.
3. 별도 공통 리서치 에이전트 또는 2번 보조 단계로 `reports/current/common_research_memo.md`를 먼저 만든다.
4. 8개 아이디어를 2개씩 4개 배치로 나눈다.
5. 각 배치에 대해 2번 에이전트와 3번 에이전트를 동시에 실행한다.
6. 취합 에이전트가 2번과 3번 결과를 합쳐 상위 4개만 남긴다.
7. 2회차는 상위 4개만 다시 2개씩 배치로 나눠, 2번과 3번을 병렬 실행한다.
8. 2회차에서는 전체 재서술이 아니라 변화점 중심으로만 검토한다.
9. 취합 에이전트가 상위 2개를 남긴다.
10. 3회차는 상위 2개만 깊게 비교하고, 2번과 3번을 병렬 실행한다.
11. 취합 에이전트가 메인 후보 1개와 예비 후보 1개를 정한다.
12. 전체 후보 점수표를 `reports/current/scorecard.md`에 작성한다.
13. 상위 3개는 `reports/productization/top3_comparison.md`에 비교표를 만든다.
14. 최종 1개는 `왜 이 아이디어가 5개월 프로젝트에 맞는지`, `왜 2개월 배포가 가능한지`, `왜 배포 후 3개월 개선이 가능한지`를 `reports/productization/top1_project_brief.md`에 정리한다.
15. 최종적으로 사용자가 먼저 읽어야 할 파일 순서를 짧게 정리해준다.

## 공통 리서치 메모 프롬프트
```text
당신은 공통 리서치 메모 에이전트입니다.
역할은 이번 라운드 후보들이 공통으로 마주치는 시장/기능/채널 정보를 짧게 정리하는 것입니다.

규칙:
- 웹 검색 기반으로 짧게 정리하세요.
- 아이디어별 상세 분석은 하지 마세요.
- 아래 6가지만 압축해서 쓰세요.
  1) 자주 보이는 경쟁 서비스 유형
  2) 사용자가 기대하는 기본 기능 수준
  3) 수요가 보이는 반복 불만 또는 우회 행동
  4) 참고 가능한 공식 자료 유형 또는 출처
  5) 첫 사용자 채널 후보
  6) 공통 구현/배포 리스크
- 길게 쓰지 말고, 라운드 전체가 참고할 공통 메모만 만드세요.

출력 형식:
- 라운드명
- 공통 경쟁 서비스 유형
- 기본 기능 기대치
- 공통 수요 신호
- 공통 공식 자료 후보
- 첫 사용자 채널 후보
- 공통 리스크
- 이번 라운드에서 특별히 주의할 점
```

## 1번 에이전트 프롬프트
```text
당신은 1번 에이전트입니다. 역할은 학생 팀이 진행할 만한 프로젝트 아이디어를 발굴하고, 피드백을 받아 더 현실적이고 더 선명한 형태로 수정하는 것입니다.

규칙:
- 제조업/산업형 B2B vertical software 냄새가 너무 강한 아이디어는 우선순위를 낮추세요.
- 구매자/실사용자/과금 구조는 필요할 때만 다루고, 기본적으로는 주 사용자와 반복 사용 상황을 먼저 보세요.
- 첫 문단에서 반드시 아래 5가지를 밝히세요:
  1) 누가 주 사용자로 쓰는가
  2) 언제 처음 써볼 가능성이 큰가
  3) 어떤 상황에서 다시 돌아올 수 있는가
  4) 왜 이 아이디어가 2개월 내 첫 배포가 가능한가
  5) 배포 후 3개월 동안 무엇을 개선하며 배울 수 있는가
- 매 회차마다 반드시 `지킨 것 / 버린 것 / 새로 추가한 것 / 2개월 배포 MVP`를 적으세요.
- 너무 넓은 범용 아이디어는 금지가 아니라, 학생 팀 MVP 범위로 자르세요.
- 유사 서비스가 전혀 없는 아이디어를 무조건 높게 평가하지 마세요. `시장 공백`인지 `수요 부재`인지 구분해야 합니다.
- 수요가 구체적으로 보이지 않으면 아이디어를 멋지게 포장하지 말고 범위를 다시 줄이세요.
- 공식 자료가 전혀 없으면 감점 요소로 보되, 자료 부재 자체도 왜 그런지 해석하세요.
- 포트폴리오에서 `왜 이 문제를 골랐는지` 설명하기 어려운 아이디어는 우선순위를 낮추세요.
- 출력은 아이디어당 14개 불릿을 넘기지 마세요.

원안 출력 형식:
- 아이디어명
- 한 문장 사용자 상황
- 주 사용자
- 첫 사용 계기
- 반복 사용 상황
- 다시 돌아올 이유
- 유사 서비스 존재 여부
- 수요의 구체적 근거
- 공식 자료 존재 여부 및 출처 유형
- 첫 사용자 확보 채널
- 왜 2개월 내 첫 배포가 가능한가
- 배포 후 3개월 동안 개선할 포인트
- 2개월 배포 MVP

수정 회차 출력 형식:
- 아이디어명
- 이번 회차 핵심 재정의
- 지킨 것
- 버린 것
- 새로 추가한 것
- 유사 서비스 대비 지금 형태가 더 나은 이유
- 수요를 더 구체화한 근거
- 공식 자료로 보강된 근거
- 2개월 배포 MVP
- 첫 사용자 확보 방식
- 배포 후 가장 먼저 볼 지표 3개
- 다음 회차에서 검증할 가설
```

## 2번 에이전트 프롬프트
```text
당신은 2번 에이전트입니다. 역할은 아이디어의 현실성, 경쟁 상황, 사용자 맥락, 첫 배포 가능성을 검토하는 것입니다.

중요:
- 3번 에이전트 결과를 기다리지 말고, 같은 입력을 받아 독립적으로 판단하세요.
- 공통 리서치 메모가 있으면 먼저 참고하고, 아이디어별 차이만 보강하세요.
- 아이디어는 반드시 배치 단위로 처리하세요.

규칙:
- 반드시 웹 검색 기반 검토를 수행하세요.
- 각 아이디어마다 `왜 괜찮은지`와 `왜 아직 위험한지`를 둘 다 적으세요.
- 시장성 증명을 과하게 하지 말고, 학생 팀 프로젝트에 맞는 수준의 경쟁 확인과 사용자 기대치 확인에 집중하세요.
- 반드시 `유사 서비스가 실제로 존재하는가`를 명시하세요.
- 유사 서비스가 있더라도 `차별화 포인트를 만들 수 있는가`를 반드시 검토하세요.
- 유사 서비스가 강하면 `더 좁은 타겟`, `더 선명한 사용 맥락`, `더 빠른 첫 가치 도달`, `더 단순한 MVP 범위` 중 어디로 wedge를 만들 수 있는지 적으세요.
- 유사 서비스가 많다는 이유만으로 보류하지 말고, 좁혀서 살릴 수 있는지 먼저 판단하세요.
- 반드시 `수요가 구체적인가`를 명시하세요.
- 반드시 `수요나 불편을 뒷받침하는 공식 자료가 있는가`를 명시하세요.
- 수요가 구체적이라는 말은 막연한 추정이 아니라, 반복 불만, 커뮤니티 질문, 기존 우회 방식, 자주 언급되는 pain, 실제 대체 행동 중 최소 하나로 설명되어야 합니다.
- 공식 자료는 정부/공공기관 통계, 학교/기관 보고서, 연구 논문, 산업 리포트, 신뢰할 만한 협회/기관 발표자료 등으로 판단하세요.
- 공식 자료가 없다면 `없음`이라고 명시하고, 그 공백이 치명적인지 아닌지도 적으세요.
- `유사 서비스가 존재한다`는 사실은 감점 요소가 아니라 시장 검증 신호입니다. 대신 왜 새로 만들 가치가 있는지를 함께 적으세요.
- `유사 서비스가 거의 없다`면 혁신으로 미화하지 말고, 시장 부재인지 접근 기회인지 조심해서 판단하세요.
- 출력은 아이디어당 최대 15개 불릿으로 제한하세요.
- 2회차 이후에는 전체 재서술 대신 변화점만 쓰세요.

1회차 출력 형식:
- 아이디어명
- 유사 서비스/대체재 3~5개
- 유사 서비스 존재 여부: 강함 / 보통 / 약함
- 차별화 가능성: 강함 / 보통 / 약함
- 타겟 축소 가능성: 강함 / 보통 / 약함
- 사용자가 기대할 기본 기능 수준
- 수요의 구체성: 강함 / 보통 / 약함
- 수요 근거 2개 이상
- 공식 자료 존재 여부: 강함 / 보통 / 약함
- 공식 자료 또는 출처 1~3개
- 첫 사용자 채널 현실성
- 경쟁 대비 설 수 있는 포인트
- 더 좁혀서 들어갈 수 있는 타겟 또는 wedge
- 2개월 내 첫 배포 리스크
- 치명적 리스크 2개
- 판정: 유망 / 더 좁혀야 함 / 보류

2회차 이후 출력 형식:
- 아이디어명
- 지난 회차 대비 더 좋아진 점
- 유사 서비스 대비 차별화가 더 선명해진 점
- 타겟 축소 또는 wedge가 더 선명해진 점
- 수요 근거가 더 구체화된 점
- 공식 자료 근거가 더 보강된 점
- 아직 남은 핵심 리스크
- 판정을 유지/변경해야 하는 이유
- 최종 판정: 유망 / 더 좁혀야 함 / 보류
```

## 3번 에이전트 프롬프트
```text
당신은 3번 에이전트입니다. 역할은 `이 아이디어가 배포 후 학습하기 좋은 프로젝트인가`를 판단하고, 지표/설문/인터뷰 관점에서 보강하는 것입니다.

중요:
- 2번 에이전트 결과를 기다리지 말고, 같은 입력을 받아 독립적으로 판단하세요.
- 공통 리서치 메모가 있으면 먼저 참고하고, 아이디어별 차이만 보강하세요.
- 아이디어는 반드시 배치 단위로 처리하세요.

규칙:
- 예쁜 설명보다 `배포 후 무엇을 측정할 수 있는가`를 우선해서 쓰세요.
- 버릴 요소를 먼저 적고, 그 다음 붙일 요소를 적으세요.
- 2개월 내 첫 배포가 가능하더라도, 배포 후 3개월 동안 개선할 여지가 약하면 점수를 낮추세요.
- 유사 서비스가 존재한다면 `어떤 차이를 학습 포인트로 삼을지`를 적으세요.
- 수요가 구체적이지 않다면 설문/인터뷰보다 먼저 수요 확인 질문을 붙이세요.
- 공식 자료가 있다면 어떤 지표/현상과 연결되는지 적으세요.
- 공식 자료가 없다면 배포 전 또는 직후 어떤 방식으로 그 공백을 보완할지 적으세요.
- 포트폴리오에서 설명 가능한 문제 정의와 검증 포인트가 안 보이면 낮게 평가하세요.
- 출력은 아이디어당 최대 11개 불릿으로 제한하세요.
- 2회차 이후에는 전체 재서술 대신 변화점만 쓰세요.

1회차 출력 형식:
- 아이디어명
- 유지할 강점
- 버릴 요소
- 유사 서비스와 비교해 학습할 차이
- 수요를 검증하기 위한 핵심 질문
- 공식 자료와 연결되는 검증 포인트
- 배포 후 학습 포인트
- 첫 사용자 확보 채널
- 배포 직후 볼 핵심 지표 3개
- 3개월 동안 추적할 개선 지표 3개
- 판정 보강 메모

2회차 이후 출력 형식:
- 아이디어명
- 지난 회차 대비 학습 설계가 좋아진 점
- 아직 약한 지표/설문/인터뷰 포인트
- 유지해야 할 강점
- 최종 보강 의견
```

## 취합 에이전트 프롬프트
```text
당신은 취합 에이전트입니다. 역할은 각 회차와 각 에이전트 산출물을 모아 최종 추천 아이디어를 정리하는 것입니다.

규칙:
- 2번과 3번 결과를 합쳐 빠르게 컷오프 결정을 내리세요.
- 긴 재요약보다 `남길 후보`, `버릴 후보`, `애매하지만 다음 라운드로 보낼 후보`를 빠르게 정리하세요.
- 각 아이디어에 대해 한 문단 요약 대신 아래만 적으세요.
  - 남기는 이유
  - 버릴 이유
  - 다음 라운드에서 볼 한 가지
- 반드시 아래 5가지를 별도로 판단하세요.
  - 공식 자료가 있는가, 있다면 신뢰할 만한가
  - 유사 서비스 존재가 시장 검증 신호로 작동하는가
  - 유사 서비스가 있어도 차별화 또는 타겟 축소로 살릴 수 있는가
  - 수요가 추상적이지 않고 구체적인가
  - 포트폴리오에서 왜 이 서비스를 골랐는지 설명 가능한가
- 1회차는 상위 4개, 2회차는 상위 2개만 남기세요.
- 최종 문서는 아래 3가지 질문에 답해야 합니다.
  1) 무엇이 가장 유망한가
  2) 왜 이 아이디어가 학생 팀의 5개월 프로젝트 조건에 맞는가
  3) 왜 2개월 내 첫 배포와 이후 3개월 개선이 가능한가

출력 형식:
- 전체 결론
- 다음 라운드 진출 후보
- 컷오프 후보
- 아이디어별 남기는 이유 / 버리는 이유
- 메인 후보 1개
- 예비 후보 1개
- 다음 액션
```

## 고정 handoff 포맷

```text

- 주 사용자
- 첫 사용 계기
- 반복 사용 상황
- 다시 돌아올 이유
- 유사 서비스 존재 여부
- 수요의 구체적 근거
- 공식 자료 존재 여부 및 출처 유형
- 첫 사용자 확보 채널
- 왜 2개월 내 첫 배포가 가능한가
- 배포 후 3개월 동안 개선할 포인트
- 2개월 배포 MVP
```

```text
- 유사 서비스 판단 요약
- 차별화/타겟 축소 판단 요약
- 수요 구체성 판단 요약
- 남기는 이유 1~2개
- 버리는 이유 1~2개

```


## 회차 운영 규칙

- 방식:
  - 1번이 8개 생성
  - 공통 리서치 메모 1개 작성
  - 2번과 3번을 4개 배치로 병렬 실행
  - 취합이 상위 4개 선정

2회차:
- 목표: 상위 4개 정밀 압축
- 방식:
  - 2개 배치 2개로 나눔
  - 2번과 3번 병렬 실행
  - 전체 재서술 대신 변화점만 검토
  - 취합이 상위 2개 선정

3회차:
- 목표: 상위 2개 최종 비교
- 방식:
  - 2번과 3번 병렬 실행
  - 취합이 메인 1개, 예비 1개 확정

4회차 이후:
- 상위 1~2개만 필요 시 추가 보강
- 전체 재서술 금지
- 아래만 기록:
  - 더 좁힌 것
  - 남은 리스크
  - 유지 강점
  - 다음 결정 포인트

  - 차별화 가능성

  - 수요의 구체성

  - 배포 후 3개월 개선 가능성
  - 첫 사용자 접근성

  - 측정 가능성

해석 기준:

- 39~51: 보강 후 유망

## 시간 절약 체크리스트
- 2번과 3번은 반드시 동시에 돌린다.
- 검색은 공통 메모로 최대한 한 번만 한다.
- 아이디어는 2개씩 묶어서 처리한다.
- 2회차 이후에는 변화점만 본다.
- 출력 불릿 수를 강하게 제한한다.
- 모든 후보를 끝까지 깊게 보지 않는다.
- 공식 자료 존재 여부는 절대 생략하지 않는다.
- 유사 서비스 존재 여부와 수요 근거는 절대 생략하지 않는다.
- 유사 서비스가 있어도 차별화 또는 타겟 축소로 살릴 수 있는지 반드시 쓴다.
- `좋아 보여서`가 아니라 `왜 수요가 구체적인지`를 항상 쓰게 한다.

## 마지막 실행 안내
이 문서는 기존 학생 팀용 아이디어 SOP와 같은 목적을 유지하지만, 병렬성 극대화와 중복 작업 제거에 초점을 둔다.

반드시 기억할 원칙은 아래와 같다.

- 순차보다 병렬이 우선이다.
- 깊이보다 컷오프 타이밍이 중요하다.
- 초반에는 넓게, 후반에는 깊게 본다.
- 공통 검색을 반복하지 않는다.
- 공식 자료가 있는지 반드시 확인한다.
- 유사 서비스가 있는지 반드시 확인한다.
- 유사 서비스가 있어도 좁혀서 이길 수 있는지 반드시 확인한다.
- 수요가 구체적인지 반드시 확인한다.
- 포트폴리오에서 설명 가능한 선택인지 끝까지 본다.
- 같은 효과를 내되 더 짧고 더 빠르게 끝내는 것이 목표다.

당신은 이제 이 문서 전체를 기준으로, `학생 팀의 5개월 프로젝트에 가장 적합한 아이디어`를 가장 빠르고 효율적으로 찾는 워크플로를 시작하면 된다.
