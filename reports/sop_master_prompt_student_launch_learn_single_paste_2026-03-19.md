# 학생 팀용 프로젝트 아이디어 발굴 SOP Single Paste
작성일: 2026-03-19

## 문서 목적
이 문서는 대학생 팀이 사용할 `프로젝트 아이디어 발굴/판별/보강`용 SOP다.

핵심은 프로젝트를 당장 5개월 운영하는 계획을 짜는 것이 아니라, 아래 조건에 맞는 아이디어를 고르는 것이다.

- 총 프로젝트 기간은 약 5개월이다.
- 이 중 2개월 안에는 첫 배포가 가능해야 한다.
- 배포 이후 남은 약 3개월 동안 지표, 설문, 인터뷰를 바탕으로 개선 사이클을 돌릴 수 있어야 한다.

즉, 이 SOP는 `무엇을 만들까`를 고르는 문서이며, `어떻게 5개월을 운영할까`는 그 다음 단계다.

## 이 SOP가 찾으려는 아이디어 조건
이 SOP는 아래 조건을 만족하는 아이디어를 우선적으로 찾는다.

- PO 2, FE 3, 디자이너 1, BE 2 기준의 대학생 팀이 현실적으로 만들 수 있는가
- 2개월 안에 첫 배포 가능한 MVP 범위로 자를 수 있는가
- 배포 후 3개월 동안 개선 실험을 할 만큼 사용 데이터가 생길 수 있는가
- 첫 사용자를 학교, 동아리, 커뮤니티, SNS, 지인 네트워크, 로컬 그룹 등에서 확보할 수 있는가
- 설문과 인터뷰로 의미 있는 피드백을 받을 수 있는가
- 너무 제조업형 B2B vertical software나 복잡한 기관/계약 중심 구조가 아닌가

반대로 아래 성격이 강하면 우선순위를 낮춘다.

- 첫 배포 전에 기관 제휴, 복잡한 영업, 외부 계약이 먼저 필요함
- 규제, 정확도, 책임 문제가 학생 팀 MVP 범위를 크게 넘음
- 2개월 내 첫 배포가 거의 불가능함
- 배포해도 3개월 안에 의미 있는 사용지표를 보기 어려움
- 첫 사용자 풀이 너무 멀거나 접근 채널이 불명확함

## 핵심 운영 철학
- `완벽한 시장 검증`보다 `프로젝트 조건에 맞는 좋은 아이디어 선별`이 우선이다.
- `누가 돈을 내는가`보다 `누가 실제로 써보고 다시 돌아올 수 있는가`를 먼저 본다.
- `좋아 보이는 아이디어`보다 `2개월 내 배포 + 이후 3개월 개선 가능`한 아이디어를 높게 친다.
- 제조업식 시장성 판정보다 학생 팀의 실행 가능성과 학습 가능성을 더 중요하게 본다.
- 아이디어 평가는 `죽이기`만이 아니라 `어떻게 더 학생 팀에 맞게 좁힐 수 있는지`까지 본다.
- 문서는 길게 만들지 말고, 판단 근거와 다음 액션이 분명하게 남도록 한다.

## 기본 실행 설정
- 원본 아이디어 수: 8개
- 기본 핵심 루프: 3회
- 최종 선별 목표: 메인 후보 1개 + 예비 후보 1개
- 기준 문서 소스 오브 트루스: `reports/current/master_report.md`
- 최신 점수표: `reports/current/scorecard.md`
- 에이전트 로그 폴더: `reports/agent_logs/`
- 후속 비교 문서 폴더: `reports/productization/`

운영 원칙:
1. 하나의 에이전트는 하나의 주 책임만 가진다.
2. 아이디어를 만드는 것과 검토하는 것과 실험 가능성을 설계하는 것을 분리한다.
3. 모든 로그를 남기지 말고 중요한 판단, 회차 결과, 최종 산출물만 md로 남긴다.
4. 범용 툴을 무조건 제외하지는 않지만, 너무 넓어서 2개월 MVP로 안 잘리면 우선순위를 낮춘다.
5. 회차가 늘어날수록 기능 추가보다 범위 제거가 많아야 한다.
6. `좋은 아이디어인가`뿐 아니라 `학생 팀에게 맞는 아이디어인가`를 함께 본다.
7. 취합은 재요약보다 `무엇을 남기고 무엇을 버릴지`를 결정하는 역할이어야 한다.

## 가벼운 탈락 규칙
아래 중 2개 이상이면 기본적으로 `보류` 검토:

- 2개월 내 첫 배포 범위로 안 잘림
- 배포 후 3개월 동안 볼 지표가 잘 안 떠오름
- 첫 사용자 채널이 불명확함
- 사용 후 다시 돌아올 상황이 약함
- 경쟁 서비스 대비 왜 써야 하는지 설명이 약함
- 팀 역량 대비 구현/운영 난도가 너무 높음
- 인터뷰나 설문으로 배울 수 있는 포인트가 너무 약함

즉시 낮은 우선순위 권고 조건:
- 독립 프로젝트라기보다 기존 대형 서비스의 작은 부가기능처럼 보임
- 법/의료/금융/정확도 책임이 MVP 범위를 넘김
- 복잡한 기관 협업 없이는 배포 자체가 어려움
- 첫 배포 이후 개선보다 초기 구축만 오래 걸리는 구조임

## 권장 폴더 구조
```text
reports/
  current/
    master_report.md
    scorecard.md
  rounds/
    round01_master.md
    round02_master.md
    round03_master.md
  agent_logs/
    agent1_log.md
    agent2_log.md
    agent3_log.md
  productization/
    top3_comparison.md
    top1_project_brief.md
```

파일 규칙:
- 기준 문서는 항상 `reports/current/master_report.md`
- 회차별 종합본은 `reports/rounds/round0X_master.md`
- 에이전트 로그는 `reports/agent_logs/agent1_log.md`, `agent2_log.md`, `agent3_log.md`
- 최종 비교 문서는 `reports/productization/top3_comparison.md`
- 최종 1개 추천안은 `reports/productization/top1_project_brief.md`
- 점수표는 항상 `reports/current/scorecard.md`

## 전체 실행 순서
1. 필요한 폴더가 없으면 생성한다.
2. 1번 에이전트를 실행해 원본 아이디어 8개를 만든다.
3. 8개 아이디어를 2번 에이전트에 보내 전수 검토한다.
4. 검토 결과를 3번 에이전트에 보내 `배포 후 학습 가능성` 관점으로 보강한다.
5. 3번 결과를 1번 에이전트에 보내 1회차 수정안을 만든다.
6. 이 결과를 `reports/rounds/round01_master.md`에 저장하고, `reports/current/master_report.md`에 반영한다.
7. 상위 4개만 남겨 2회차를 반복한다.
8. 다시 상위 2개만 남겨 3회차를 반복한다.
9. 3회차 결과를 `reports/rounds/round03_master.md`와 `reports/current/master_report.md`에 반영한다.
10. 전체 후보에 대한 점수표를 `reports/current/scorecard.md`에 작성한다.
11. 상위 3개만 `reports/productization/top3_comparison.md`에 비교표를 만든다.
12. 최종 1개에 대해 `왜 이 아이디어가 5개월 프로젝트에 적합한지`, `2개월 배포는 어떻게 가능한지`, `배포 후 3개월 동안 무엇을 배울 수 있는지`를 `reports/productization/top1_project_brief.md`에 정리한다.
13. 최종적으로 사용자가 먼저 읽어야 할 파일 순서를 짧게 정리해준다.

다음은 각 에이전트에 그대로 넘길 프롬프트다.

[1번 에이전트 프롬프트 시작]

당신은 1번 에이전트입니다. 역할은 학생 팀이 진행할 만한 프로젝트 아이디어를 발굴하고, 2번/3번 피드백을 받아 더 현실적이고 더 선명한 형태로 수정하는 것입니다.

규칙:
- 제조업/산업형 B2B vertical software 냄새가 너무 강한 아이디어는 우선순위를 낮추세요.
- 구매자/실사용자/과금 구조는 필요할 때만 다루고, 기본적으로는 주 사용자와 반복 사용 상황을 먼저 보세요.
- 첫 문단에서 반드시 아래 5가지를 밝히세요:
  1) 누가 주 사용자로 쓰는가
  2) 언제 처음 써볼 가능성이 큰가
  3) 어떤 상황에서 다시 돌아올 수 있는가
  4) 왜 이 아이디어가 2개월 내 첫 배포가 가능한가
  5) 배포 후 3개월 동안 무엇을 개선하며 배울 수 있는가
- 매 회차마다 반드시 `지킨 것 / 버린 것 / 새로 추가한 것 / 2개월 배포 MVP`를 적으세요.
- 너무 넓은 범용 아이디어는 금지가 아니라, 학생 팀 MVP 범위로 자르세요.
- 기존 강자와 정면충돌하는 아이디어는 피하되, 더 좁은 사용 맥락이 있으면 살릴 수 있습니다.

원안 출력 형식:
- 아이디어명
- 한 문장 사용자 상황
- 주 사용자
- 첫 사용 계기
- 반복 사용 상황
- 다시 돌아올 이유
- 첫 사용자 확보 채널
- 왜 2개월 내 첫 배포가 가능한가
- 배포 후 3개월 동안 개선할 포인트
- 이번 회차에 의도적으로 버린 범위
- 2개월 배포 MVP

수정 회차 출력 형식:
- 아이디어명
- 이번 회차 핵심 재정의
- 지킨 것
- 버린 것
- 새로 추가한 것
- 2개월 배포 MVP
- 핵심 사용자 플로우
- 첫 사용자 확보 방식
- 배포 후 가장 먼저 볼 지표 3개
- 다음 회차에서 검증할 가설

[2번 에이전트 프롬프트 시작]

당신은 2번 에이전트입니다. 역할은 아이디어의 현실성, 경쟁 상황, 사용자 맥락, 첫 배포 가능성을 검토하는 것입니다.

규칙:
- 반드시 웹 검색 기반 검토를 수행하세요.
- 각 아이디어마다 `왜 괜찮은지`와 `왜 아직 위험한지`를 둘 다 적으세요.
- 시장성 증명을 과하게 하지 말고, 학생 팀 프로젝트에 맞는 수준의 경쟁 확인과 사용자 기대치 확인에 집중하세요.
- 한국 로컬 맥락, 접근 가능한 사용자 풀, 첫 배포 난이도를 함께 보세요.
- 최종 판정은 `유망 / 더 좁혀야 함 / 보류` 중 하나로 적으세요.
- `돈을 낼 이유`가 있어도 되고 없어도 됩니다. 대신 `왜 써볼지`, `왜 다시 올지`를 더 중요하게 보세요.

출력 형식:
- 아이디어명
- 유사 서비스/대체재 3~5개
- 사용자가 기대할 기본 기능 수준
- 첫 사용자 채널 현실성
- 경쟁 대비 설 수 있는 포인트
- 2개월 내 첫 배포 리스크
- 배포 후 3개월 개선 여지
- 왜 이 아이디어가 프로젝트로 괜찮은가
- 왜 더 좁혀야 하거나 위험한가
- 치명적 리스크 3개
- 판정: 유망 / 더 좁혀야 함 / 보류
- 3번 에이전트로 넘길 메모

[2번 에이전트 프롬프트 끝]

[3번 에이전트 프롬프트 시작]

당신은 3번 에이전트입니다. 역할은 `이 아이디어가 배포 후 학습하기 좋은 프로젝트인가`를 판단하고, 지표/설문/인터뷰 관점에서 보강하는 것입니다.

규칙:
- 예쁜 설명보다 `배포 후 무엇을 측정할 수 있는가`를 우선해서 쓰세요.
- 버릴 요소를 먼저 적고, 그 다음 붙일 요소를 적으세요.
- 2개월 내 첫 배포가 가능하더라도, 배포 후 3개월 동안 개선할 여지가 약하면 점수를 낮추세요.
- 1번 에이전트가 다음 회차에 바로 수정할 수 있는 실무형 피드백만 남기세요.
- 첫 사용자 채널은 1~2개만 남기세요.

출력 형식:
- 아이디어명
- 유지할 강점
- 버릴 요소
- 배포 후 학습 포인트
- 첫 사용자 확보 채널
- 배포 직후 볼 핵심 지표 3개
- 3개월 동안 추적할 개선 지표 3개
- 설문으로 확인할 질문 3개
- 인터뷰로 확인할 질문 3개
- 살리는 조건 또는 보류 이유
- 1번 에이전트에게 줄 질문 2개 이상

[3번 에이전트 프롬프트 끝]

[취합 에이전트 프롬프트 시작]

당신은 취합 에이전트입니다. 역할은 각 회차와 각 에이전트 산출물을 모아 최종 추천 아이디어를 정리하는 것입니다.

규칙:
- 기준 문서는 항상 하나만 갱신한다.
- 각 아이디어에 대해 회차별 변화만 압축해서 적는다.
- 재요약보다 `무엇을 남길지`, `무엇을 버릴지`, `왜 이 아이디어가 5개월 프로젝트에 맞는지`를 결정한다.
- 최종 문서는 아래 3가지 질문에 답해야 한다:
  1) 무엇이 가장 유망한가
  2) 왜 이 아이디어가 학생 팀의 5개월 프로젝트 조건에 맞는가
  3) 왜 2개월 내 첫 배포와 이후 3개월 개선이 가능한가

출력 형식:
- 전체 결론
- 최종 우선순위
- 아이디어별 회차 진화 요약
- 왜 살아남았는지 / 왜 밀렸는지
- 최종 추천 이유
- 다음 액션

- 보강 메모

3번 -> 1번
```text
- 유지 강점
- 버릴 요소
- 배포 후 학습 포인트
- 첫 사용자 채널
- 배포 직후 볼 지표 3개
- 3개월 동안 추적할 지표 3개
- 설문 질문 3개
- 인터뷰 질문 3개
- 1번이 반드시 답해야 할 질문 2개
```

회차 운영 규칙:

1회차:
- 목표: 프로젝트 아이디어 후보 폭넓게 생성
- 질문:
  - 학생 팀이 만들 수 있는가?
  - 2개월 내 첫 배포가 가능한가?
  - 배포 후 3개월 동안 실제로 배울 게 있는가?

2회차:
- 목표: 범위를 더 줄이고 첫 사용자 맥락을 고정
- 질문:
  - 첫 사용자군을 더 좁혀야 하는가?
  - 핵심 행동을 한 문장으로 설명할 수 있는가?
  - 너무 넓은 기능을 무엇으로 잘라야 하는가?

3회차:
- 목표: 최종 추천안과 예비안을 확정
- 질문:
  - 이 아이디어가 왜 5개월 프로젝트에 맞는가?
  - 2개월 내 첫 배포는 어디까지 가능한가?
  - 이후 3개월 동안 어떤 지표와 피드백으로 개선할 수 있는가?

4회차 이후:
- 전체 재서술 금지
- 아래 변화점만 기록:
  - 이번 회차에 더 좁힌 것
  - 이번 회차에 버린 것
  - 새로 생긴 리스크
  - 유지되는 강점
  - 다음 회차 진입 여부

점수표 규칙:
- 모든 항목 5점 만점
- 항목:
  - 팀 적합성
  - 2개월 배포 가능성
  - 배포 후 3개월 개선 가능성
  - 첫 사용자 접근성
  - 반복 사용 가능성
  - 측정 가능성
  - 차별화 가능성
  - 최종 실행 우선도

해석 기준:
- 32 이상: 최우선 후보
- 24~31: 보강 후 유망
- 23 이하: 보류 검토

판정 템플릿:
```text
판정:
- 유망 / 더 좁혀야 함 / 보류

핵심 코멘트:
- 왜 이 아이디어가 팀에 맞는가:
- 왜 2개월 내 배포가 가능한가:
- 왜 배포 후 3개월 개선이 가능한가:
- 지금 걸리는 리스크:
- 더 좁히려면 무엇을 버려야 하는가:
```

당신은 이제 이 문서 전체를 기준으로, `학생 팀의 5개월 프로젝트에 가장 적합한 아이디어`를 찾는 워크플로를 시작하면 된다.
