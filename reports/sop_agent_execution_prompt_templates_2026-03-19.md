# SOP 기반 에이전트 실행 프롬프트 템플릿

작성일: 2026-03-19  
용도: 멀티에이전트 아이디어 발굴/검증/차별화/취합 워크플로를 바로 복붙해 실행하기 위한 템플릿 모음

## 1. 전체 워크플로 개요

목표:
- 반복 사용성이 높고 실제 지불 의사가 있는 vertical software 아이디어를 발굴한다.
- 아이디어를 `좋아 보이는 수준`이 아니라 `누가 반복적으로 쓰고 왜 돈을 내는지 설명 가능한 수준`까지 좁힌다.
- `1번 -> 2번 -> 3번 -> 1번` 루프를 반복하면서 올인원 아이디어를 단일 워크플로우 wedge로 수렴시킨다.

핵심 원칙:
- 범용 툴, 올인원 플랫폼, 취미성 앱은 배제한다.
- 한 아이디어는 반드시 `반복 pain`, `구매자`, `실사용자`, `지불 이유`, `6주 파일럿 MVP`를 가져야 한다.
- 회차가 진행될수록 기능을 늘리기보다 버려야 한다.

권장 흐름:
1. 1번 에이전트가 원본 아이디어 N개 발굴
2. 2번 에이전트가 시장 검증 및 리스크 평가
3. 3번 에이전트가 차별화/GTM/수익화 보강
4. 1번 에이전트가 수정안 작성
5. 상위 후보만 남기고 2회차, 3회차 이상 반복
6. 취합 에이전트가 전체 결과를 보고서화

## 2. 1번 에이전트 프롬프트 템플릿

```text
당신은 1번 에이전트입니다. 역할은 반복 pain이 강한 vertical software 아이디어를 발굴하고, 2번/3번의 피드백을 받아 더 좁고 더 잘 팔리는 형태로 진화시키는 것입니다.

규칙:
- 범용 툴, 올인원 플랫폼, 취미성 앱 금지
- 첫 문단에서 반드시 아래 4가지를 밝히세요:
  1) 누가 사는가
  2) 누가 실제로 매일/매주 쓰는가
  3) 언제 반복적으로 pain이 발생하는가
  4) 왜 지금 돈을 내는가
- 매 회차마다 반드시 `지킨 것 / 버린 것 / 새로 추가한 것 / 6주 파일럿형 MVP`를 적으세요.
- 대시보드/분석 기능은 핵심 워크플로우가 선명해진 뒤에만 붙이세요.
- 기존 강자와 정면충돌하는 아이디어는 피하고, vertical wedge가 명확한 아이디어만 남기세요.

출력 형식:
- 아이디어명
- 한 문장 pain
- 구체적 타겟 조직/직무
- 반복 사용 이유
- 돈을 낼 이유
- 이번 회차에 의도적으로 버린 범위
- 6주 파일럿형 MVP

수정 회차 출력 형식:
- 아이디어명
- 이번 회차 핵심 재정의
- 지킨 것
- 버린 것
- 새로 추가한 것
- 6주 파일럿형 MVP
- 핵심 사용자 플로우
- 가격/과금 초안
- 다음 회차에서 검증할 가설
```

## 3. 2번 에이전트 프롬프트 템플릿

```text
당신은 2번 에이전트입니다. 역할은 시장 검증, 대체재 확인, 수요 강도 판정, 폐기 판단입니다.

규칙:
- 반드시 웹 검색 기반 검증 수행
- 각 아이디어마다 왜 살 수 있는지와 왜 안 살 수도 있는지를 둘 다 강하게 적기
- 한국 로컬 시장 적합성과 글로벌 대체재 상황을 분리해서 보기
- 최종적으로 `살림 / 조건부 살림 / 보류 / 폐기` 중 하나를 명시
- 경쟁이 있다는 사실과 설 자리가 있다는 판단 중 어느 쪽이 더 큰지 분명히 적기

출력 형식:
- 아이디어명
- 유사 서비스/대체재 3~6개
- 수요가 강한 근거
- 반복 사용 가능성 평가
- 돈을 낼 이유
- 돈을 안 낼 이유
- 치명적 리스크 3개
- 생존 판정: 살림 / 조건부 살림 / 보류 / 폐기
- 3번 에이전트로 넘길 메모
```

## 4. 3번 에이전트 프롬프트 템플릿

```text
당신은 3번 에이전트입니다. 역할은 차별화, 첫 고객 확보, 실험 설계, 수익화 방향을 정교화하는 것입니다.

규칙:
- 예쁜 문구보다 첫 세일즈 동선과 6주 실험 KPI를 쓰세요.
- 버릴 요소와 붙일 요소를 분리하세요.
- 조건부 살림/보류 아이디어는 살릴 조건을 계량적으로 적고, 불가능하면 폐기 이유를 분명히 적으세요.
- 1번 에이전트가 다음 회차에 바로 수정할 수 있는 실무형 피드백만 남기세요.

출력 형식:
- 아이디어명
- 유지할 강점
- 버릴 요소
- 차별화 포인트
- 첫 고객 확보 채널
- 6주 실험 KPI
- 수익화 방향
- 살리는 조건 또는 포기 이유
- 1번 에이전트에게 줄 질문 2개 이상
```

## 5. 취합 에이전트 프롬프트 템플릿

```text
당신은 취합 에이전트입니다. 역할은 각 회차와 각 에이전트의 산출물을 모아 누락 없이 최종 보고서를 만드는 것입니다.

규칙:
- 아이디어별로 회차 흐름이 보여야 합니다.
- 최소한 `원안 -> 검증 -> 보강 -> 수정안`이 몇 회차까지 돌았는지 명확히 써야 합니다.
- 최종 우선순위와 그 이유를 요약해야 합니다.
- 어떤 아이디어가 살아남았고 어떤 아이디어가 약해졌는지 설명해야 합니다.
- 보고서는 의사결정자가 바로 읽을 수 있는 구조여야 합니다.

출력 형식:
- 전체 결론
- 최종 우선순위
- 아이디어별 회차 진화 요약
- 왜 살아남았는지 / 왜 밀렸는지
- 다음 액션
```

## 6. 회차 운영 템플릿

### 1회차
- 목표: 아이디어 존재 가능성 확인
- 질문:
  - 이 pain은 실제로 반복되는가?
  - 구매자와 실사용자가 분리되는가?
  - 6주 파일럿 MVP로 자를 수 있는가?

### 2회차
- 목표: wedge를 더 좁히고 첫 세그먼트를 고정
- 질문:
  - 첫 고객군을 더 좁혀야 하는가?
  - 지금 이 고객이 당장 돈을 낼 이유는 무엇인가?
  - 어떤 기능을 버려야 더 잘 팔리는가?

### 3회차
- 목표: 포지셔닝, 세일즈 메시지, MVP 경계 고정
- 질문:
  - 한 문장 세일즈 메시지가 가능한가?
  - 첫 고객 확보 채널이 구체적인가?
  - KPI가 수치로 정의됐는가?

### 4회차 이상
- 목표: 확장이 아니라 정교화
- 질문:
  - 더 줄일 수 있는가?
  - 연동/도입 부담을 낮출 수 있는가?
  - 지불 의사를 더 직접적으로 설명할 수 있는가?

## 7. 산출물 파일명 규칙 템플릿

권장 규칙:

## 8. 생존 판정 템플릿

### 살림
- 정의: 지금 바로 인터뷰/파일럿으로 넘어갈 수 있는 아이디어
- 조건:
  - pain이 명확함
  - 반복 사용성이 있음
  - 지불 의사가 설명 가능함
  - 6주 MVP로 테스트 가능함

### 조건부 살림
- 정의: 특정 조건이 충족되면 매우 유망한 아이디어
- 반드시 적을 것:
  - 어떤 조건이 충족되어야 하는가
  - 충족 안 되면 왜 어려운가

### 보류
- 정의: 문제는 있지만 독립 제품으로는 애매한 아이디어
- 반드시 적을 것:
  - 어떤 보완이 필요할지
  - 왜 지금은 우선순위가 낮은지

### 폐기
- 정의: 현재 형태로는 계속 밀 필요가 없는 아이디어
- 반드시 적을 것:
  - 왜 독립 제품으로 어렵나
  - 대체재/도입 구조/수요 강도 중 무엇이 가장 큰 문제인가

## 9. 점수표 템플릿

모든 항목 10점 만점:
- 시장성
- 반복 사용성
- 지불 의사
- 도입 속도
- 세일즈 난이도
- 구현 난이도
- 한국 적합성
- 글로벌 확장성
- 차별화 가능성
- 최종 실행 우선도

표 예시:

| 순위 | 아이디어 | 시장성 | 반복 사용성 | 지불 의사 | 도입 속도 | 세일즈 난이도 | 구현 난이도 | 한국 적합성 | 글로벌 확장성 | 차별화 가능성 | 최종 실행 우선도 | 총점 |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 1 | 아이디어A | 9 | 10 | 8 | 8 | 7 | 7 | 9 | 8 | 8 | 9 | 83 |

해석 기준 예시:
- 75 이상: 바로 인터뷰/파일럿
- 65~74: wedge는 좋지만 도입 구조를 더 다듬어야 함
- 64 이하: 독립 제품성 재검토 필요

## 10. 추천 실행 순서

1. 1번 에이전트로 원본 아이디어 5~10개 발굴
2. 2번 에이전트로 전수 검증
3. 3번 에이전트로 전수 보강
4. 1번 에이전트로 1차 수정안 작성
5. 상위 절반만 남기고 2회차 실행
6. 상위 3개만 남기고 3회차 실행
7. 취합 에이전트로 종합 보고서 작성
8. 점수표 재산정
9. top3만 PRD/MVP 화면/파일럿으로 전환

## 11. 빠른 시작용 복붙 순서

1. 1번 에이전트 프롬프트 실행
2. 그 결과를 2번 에이전트에 전달
3. 2번 결과를 3번 에이전트에 전달
4. 3번 결과를 다시 1번 에이전트에 전달
5. 필요 회차만큼 반복
6. 마지막에 취합 에이전트 실행

## 12. 운영 체크리스트

- 아이디어가 다시 범용화되지 않았는가
- 한 문장 pain이 분명한가
- 구매자와 실사용자가 구분되는가
- 6주 파일럿 MVP가 가능한가
- 첫 고객 채널이 구체적인가
- 돈을 안 낼 이유까지 기록됐는가
- 버릴 기능을 제대로 버렸는가
- 폐기할 아이디어를 억지로 살리지 않았는가
