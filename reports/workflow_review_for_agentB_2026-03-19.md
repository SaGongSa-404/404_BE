# 워크플로 평가 보고서 for Agent B

작성일: 2026-03-19

이 문서는 지금까지 운영한 멀티에이전트 워크플로를 비판적으로 평가해서, B 에이전트가 개선 버전을 설계할 수 있도록 만든 문서다. 칭찬보다 구조적 개선 포인트를 우선한다.

## 1. 현재 워크플로 강점

1. 역할 분리가 명확하다.
   - 1번은 발굴/제품화
   - 2번은 검증/리스크 판정
   - 3번은 차별화/GTM
   - 취합은 별도

2. 아이디어를 반복 루프로 좁히는 구조가 강하다.
   - `올인원`에서 `단일 반복 워크플로우`로 수렴시키는 데 효과적이었다.

3. 산출물 중심 운영이 가능하다.
   - 회차별 결과와 에이전트별 로그가 남아 재검토와 handoff가 쉬웠다.

4. 상위 후보를 자연스럽게 압축하는 흐름이 있다.
   - 10개 -> 상위 3개 -> PRD/점수표 -> 신규 5개 실험 같은 확장 경로가 가능했다.

## 2. 구조적 약점

1. 루프가 길어질수록 중복 기록이 과도하게 늘어난다.
   - 같은 아이디어를 여러 파일에서 반복 설명하면서 정보 밀도가 떨어진다.

2. 회차 단위와 에이전트 단위 산출물이 동시에 존재해 정보 구조가 중첩된다.
   - `round 중심`, `agent 중심`, `aggregate 중심`이 같이 생기면서 어디가 소스 오브 트루스인지 흐려진다.

3. handoff 형식이 충분히 표준화되지 않았다.
   - 에이전트마다 문체와 필드가 조금씩 달라져 다음 단계가 읽고 흡수하는 비용이 높다.

4. 폐기 기준보다 개선 기준이 더 강하게 작동한다.
   - 조건부 살림이 많아지고 아이디어가 계속 살아남는 경향이 있다.

5. 취합 단계가 너무 늦게 강해진다.
   - 초기부터 어떤 파일이 기준 문서인지 명확히 고정하지 않으면 중간 산출물이 난립한다.

## 3. 병목 구간

1. `1번 -> 2번` handoff
   - 아이디어 설명이 길수록 2번이 핵심 wedge를 빠르게 파악하기 어려워진다.

2. `2번 -> 3번` handoff
   - 검증 결과가 많아질수록 3번이 핵심 위험만 뽑아 전략으로 바꾸는 비용이 커진다.

3. `3번 -> 1번` handoff
   - 전략 제안이 많아질수록 1번이 무엇을 실제 제품 범위에 반영해야 하는지 판단이 무거워진다.

4. `다회차 확장`
   - 3회차 이후 10회차로 갈 때는 새 학습보다 표현 반복이 증가할 위험이 크다.

## 4. 중복/낭비 가능성

1. 동일 아이디어 요약이 여러 파일에 반복된다.
   - 종합본, 에이전트별 로그, 점수표, 후속 PRD에서 같은 요약이 재서술된다.

2. 모든 중간 상태를 파일로 남기면 정보가 많아 보여도 실제 의사결정 속도는 느려진다.

3. 신규 아이디어 실험 파일이 여러 버전으로 병렬 생성될 수 있다.
   - 실제로 이름이 비슷한 파일들이 여러 개 생기면 최종본 판별이 어려워진다.

4. 에이전트 수가 늘수록 병렬성이 늘어나는 대신 취합 비용도 같이 커진다.

## 5. 에이전트 설계상 리스크

1. 1번 에이전트가 다시 넓어지는 경향
   - 전략 보강을 과잉 반영하면 범위가 커질 수 있다.

2. 2번 에이전트의 `조건부 살림` 과다
   - 죽여야 할 아이디어를 오래 끌고 갈 수 있다.

3. 3번 에이전트의 전략 문구 과잉
   - 실제로 제품 범위를 줄여야 하는데 GTM 문장만 늘어날 수 있다.

4. 취합 에이전트의 과도한 재요약
   - 새로운 인사이트 없이 기존 내용을 다시 정리만 할 위험이 있다.

## 6. 기록/산출물 관리 리스크

1. 파일명 규칙이 완전히 단일화되지 않았다.
   - 유사한 이름의 파일이 여러 버전 생길 수 있다.

2. 최종본/중간본 구분이 약하다.
   - 사용자가 무엇을 먼저 봐야 하는지 헷갈릴 수 있다.

3. round 기준과 agent 기준 파일이 함께 있을 때 탐색 비용이 높다.

4. 기록은 많지만 `의사결정 로그`와 `최종 산출물`이 구분되지 않으면 생산성이 떨어진다.

## 7. 우선 고쳐야 할 항목 Top 10

1. `소스 오브 트루스` 문서를 항상 하나만 지정할 것
2. handoff 필드를 완전히 표준화할 것
3. 2번 에이전트의 폐기 기준을 강화할 것
4. 1번 에이전트가 매 회차 `버린 것`을 강제하도록 할 것
5. 3번 에이전트가 `붙일 것`보다 `버릴 것`을 먼저 쓰게 할 것
6. 회차가 늘어날수록 요약 길이를 줄이고 변화점만 남길 것
7. 파일명에 `draft/final/round/agent/aggregate`를 강제해 상태를 드러낼 것
8. 신규 아이디어 실험은 항상 `1차 루프`와 `후속 루프`를 분리할 것
9. 점수표 재산정 시 같은 기준표를 고정해서 비교 가능하게 할 것
10. 취합 에이전트는 재요약보다 `결정`, `삭제`, `다음 액션`에 집중하게 할 것

## 8. B 에이전트에게 넘길 개선 요청

1. 워크플로를 `핵심 루프`, `확장 루프`, `후속 제품화` 3단 구조로 재편해달라.
2. `소스 오브 트루스`와 `참고 로그`를 분리하는 문서 체계를 만들어달라.
3. 에이전트 handoff를 위한 고정 템플릿을 설계해달라.
4. 회차가 늘어날수록 산출물이 길어지지 않고 더 짧아지도록 규칙을 넣어달라.
5. `폐기 기준`과 `계속 진행 기준`을 숫자 또는 체크리스트 수준으로 명확히 만들어달라.
6. 파일명/폴더명 규칙을 완전히 표준화해달라.
7. 사용자 입장에서 `무엇을 먼저 읽어야 하는지`가 바로 보이도록 최종 구조를 단순화해달라.
