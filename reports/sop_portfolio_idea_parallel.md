# 포트폴리오 강한 서비스 아이디어 발굴 SOP Single Paste Parallel Optimized
작성일: 2026-03-19

## 문서 목적
이 문서는 `대학생 팀이 5개월 안에 만들기에 좋고, 포트폴리오에서 강하게 설명할 수 있는 서비스 아이디어`를 찾기 위한 SOP다.

중요:
- 팀이 대학생이라는 뜻이지, 서비스의 주 사용자가 반드시 대학생이어야 한다는 뜻은 아니다.
- 목표는 `대학생 대상 서비스`를 찾는 것이 아니라, `대학생 팀이 만들기에 현실적이면서 포트폴리오 임팩트가 큰 서비스`를 찾는 것이다.
- 총 프로젝트 기간은 약 5개월이며, 이 중 2개월 안에는 첫 배포가 가능해야 한다.
- 배포 이후 남은 기간 동안 실제 사용자 피드백과 지표를 통해 개선 사이클을 보여줄 수 있어야 한다.

즉, 이 문서는 `누가 쓸 만한가`만이 아니라, 아래를 함께 설명할 수 있는 아이디어를 고른다.

- 왜 이 문제를 선택했는가
- 왜 이 팀이 이 문제를 풀기에 적합한가
- 왜 지금 이 범위로 제품을 자르는가
- 왜 유사 서비스가 있어도 새로 만들 가치가 있는가
- 왜 이 프로젝트가 포트폴리오에서 제품 감각과 실행력을 잘 보여주는가

## 이 SOP가 찾으려는 아이디어 조건
- PO 2, FE 3, 디자이너 1, BE 2 기준의 대학생 팀이 현실적으로 만들 수 있는가
- 2개월 안에 첫 배포 가능한 MVP 범위로 자를 수 있는가
- 이후 3개월 동안 실제 개선 루프를 보여줄 수 있는가
- 서비스의 주 사용자는 대학생이 아니어도 된다
- 특정 사용자 집단의 반복 문제나 반복 행동이 선명한가
- 유사 서비스 또는 대체재가 실제로 존재하는가
- 유사 서비스가 있어도 차별화 포인트 또는 더 좁은 wedge를 만들 수 있는가
- 수요가 구체적인가
- 가능하다면 공식 자료나 신뢰할 만한 근거로 문제의 존재를 보강할 수 있는가
- 포트폴리오에서 문제 정의, 제품 판단, MVP 절단, 실험, 개선 과정을 설명하기 좋은가

## 이 SOP가 피하려는 결과
아래 방향으로만 과하게 쏠리면 감점한다.

- 너무 안전하지만 재미와 제품성이 약한 체크리스트형 아이디어
- `대학생이 만들 수 있다`는 이유만으로 선택된 평범한 관리 툴
- 유사 서비스가 너무 많은데도 차별화 설명이 약한 클론형 아이디어
- 반대로 유사 서비스가 거의 없는데, 사실은 수요가 없어서 비어 있는 영역
- 포트폴리오에서 `왜 이걸 만들었는지`보다 `그냥 해봤다`에 가까운 아이디어

## 포트폴리오 강도 판단 기준
좋은 아이디어는 아래를 보여줄 수 있어야 한다.

- 문제 선택 이유가 분명하다
- 사용자와 사용 맥락이 선명하다
- MVP에서 무엇을 버렸는지 설명할 수 있다
- 유사 서비스와 무엇이 다른지 설명할 수 있다
- 첫 배포 후 어떤 데이터를 보고 어떻게 바꿀지 말할 수 있다
- 디자인, UX, 제품 구조, 실험 설계 중 최소 2개 이상에서 팀 역량을 보여줄 수 있다

## 병렬 최적화 핵심 원칙
1. 순차 의존을 줄인다.
2. 초반에는 넓고 얕게, 후반에는 좁고 깊게 본다.
3. 동일한 검색을 여러 에이전트가 반복하지 않게 한다.
4. 2회차 이후에는 전체 재서술 대신 변화점만 본다.
5. 아이디어는 `2개 묶음 배치`로 검토한다.
6. 2번과 3번 에이전트는 같은 입력을 받아 동시에 일한다.
7. 취합 에이전트는 긴 요약보다 `무엇이 포트폴리오로 강한가`를 빠르게 정리한다.

## 카테고리 다양성 규칙
1번 에이전트는 처음부터 비슷한 성격의 아이디어만 만들지 않는다.

최초 8개 아이디어는 아래처럼 최대한 섞는다.
- workflow / coordination
- creator / content / sharing
- community / matching / discovery
- personal utility with repeated behavior
- niche operational tool
- small B2B or prosumer tool
- local or campus-adjacent service
- AI-assisted but non-gimmicky service

또한 주 사용자군도 섞는다.
- 대학생
- 취준생
- 동아리/소모임 운영자
- 프리랜서/크리에이터
- 소규모 팀 리더
- 반복 행정/운영을 하는 개인 또는 소규모 집단

## 기본 실행 설정
- 원본 아이디어 수: 8개
- 기본 핵심 루프: 3회
- 최종 선별 목표: 메인 후보 1개 + 예비 후보 1개
- 검토 배치 크기: 2개 아이디어씩
- 기준 문서 소스 오브 트루스: `reports/current/master_report.md`
- 최신 점수표: `reports/current/scorecard.md`
- 공통 리서치 메모: `reports/current/common_research_memo.md`
- 에이전트 로그 폴더: `reports/agent_logs/`
- 후속 비교 문서 폴더: `reports/productization/`

## 빠른 보류 규칙
아래 중 2개 이상이면 기본적으로 `보류` 검토:

- 2개월 내 첫 배포 범위로 안 잘림
- 유사 서비스는 많지만 차별화 설명이 약함
- 유사 서비스가 거의 없고 수요 근거도 약함
- 수요가 추상적이고 반복 사용 상황이 안 보임
- 공식 자료와 비공식 근거 모두 너무 빈약함
- 첫 사용자 채널이 불명확함
- 포트폴리오에서 보여줄 제품 판단 포인트가 약함
- 팀 역량 대비 구현/운영 난도가 너무 높음

## 공통 리서치 메모 규칙
라운드마다 먼저 `공통 리서치 메모`를 만든다.

여기에는 아래만 짧게 정리한다.
- 이번 라운드 후보들이 공통으로 마주치는 경쟁 서비스 유형
- 사용자가 기대하는 기본 기능 수준
- 실제로 수요가 보이는 반복 불만, 질문, 우회 행동
- 참고 가능한 공식 자료 유형 또는 출처
- 첫 사용자 채널 후보
- 포트폴리오 관점에서 좋은 차별화 패턴
- 공통 구현/배포 리스크

## 권장 폴더 구조

```text
reports/

    agent3_batchC.md
    agent3_batchD.md
  productization/
    top3_comparison.md
    top1_project_brief.md
```

## 전체 실행 순서
1. 필요한 폴더가 없으면 생성한다.
2. 1번 에이전트를 실행해 서로 다른 카테고리의 원본 아이디어 8개를 만든다.
3. 공통 리서치 메모를 먼저 만든다.
4. 8개 아이디어를 2개씩 4개 배치로 나눈다.
5. 각 배치에 대해 2번 에이전트와 3번 에이전트를 동시에 실행한다.
6. 취합 에이전트가 2번과 3번 결과를 합쳐 상위 4개만 남긴다.
7. 2회차는 상위 4개만 다시 2개씩 배치로 나눠 병렬 검토한다.
8. 2회차에서는 전체 재서술 대신 `더 좁아진 타겟`, `더 선명한 차별화`, `더 강한 포트폴리오 설명력` 중심으로 본다.
9. 취합 에이전트가 상위 2개를 남긴다.
10. 3회차는 상위 2개만 깊게 비교한다.
11. 취합 에이전트가 메인 후보 1개와 예비 후보 1개를 정한다.
12. 전체 후보 점수표를 `reports/current/scorecard.md`에 작성한다.
13. 상위 3개는 `reports/productization/top3_comparison.md`에 비교표를 만든다.
14. 최종 1개는 `왜 이 아이디어가 팀에 맞는지`, `왜 포트폴리오로 강한지`, `왜 2개월 배포가 가능한지`, `왜 이후 개선 루프가 설계되는지`를 `reports/productization/top1_project_brief.md`에 정리한다.

## 공통 리서치 메모 프롬프트
```text
당신은 공통 리서치 메모 에이전트입니다.
역할은 이번 라운드 후보들이 공통으로 마주치는 시장/기능/채널 정보를 짧게 정리하는 것입니다.

규칙:
- 웹 검색 기반으로 짧게 정리하세요.
- 아이디어별 상세 분석은 하지 마세요.
- 아래 7가지만 압축해서 쓰세요.
  1) 자주 보이는 경쟁 서비스 유형
  2) 사용자가 기대하는 기본 기능 수준
  3) 수요가 보이는 반복 불만 또는 우회 행동
  4) 참고 가능한 공식 자료 유형 또는 출처
  5) 첫 사용자 채널 후보
  6) 포트폴리오 관점에서 좋은 차별화 패턴
  7) 공통 구현/배포 리스크

출력 형식:
- 라운드명
- 공통 경쟁 서비스 유형
- 기본 기능 기대치
- 공통 수요 신호
- 공통 공식 자료 후보
- 첫 사용자 채널 후보
- 포트폴리오 차별화 힌트
- 공통 리스크
```

## 1번 에이전트 프롬프트
```text
당신은 1번 에이전트입니다. 역할은 학생 팀이 진행할 만한 프로젝트 아이디어를 발굴하고, 피드백을 받아 더 현실적이고 더 선명한 형태로 수정하는 것입니다.

중요:
- 주 사용자가 반드시 대학생일 필요는 없습니다.
- 아이디어가 포트폴리오에서 제품 감각과 판단력을 보여줄 수 있어야 합니다.
- 처음부터 대학생 행정/체크리스트 문제로만 몰리지 마세요.

규칙:
- 제조업/산업형 B2B vertical software 냄새가 너무 강한 아이디어는 우선순위를 낮추세요.
- 범용 서비스도 가능하지만, 더 좁은 타겟이나 사용 맥락으로 자를 수 있어야 합니다.
- 아이디어 8개는 가능한 한 서로 다른 카테고리에서 만드세요.
- 첫 문단에서 반드시 아래 6가지를 밝히세요:
  1) 누가 주 사용자로 쓰는가
  2) 언제 처음 써볼 가능성이 큰가
  3) 어떤 상황에서 다시 돌아올 수 있는가
  4) 왜 이 아이디어가 2개월 내 첫 배포가 가능한가
  5) 배포 후 3개월 동안 무엇을 개선하며 배울 수 있는가
  6) 왜 이 아이디어가 포트폴리오에서 강한가
- 유사 서비스가 전혀 없는 아이디어를 무조건 높게 평가하지 마세요.
- 수요가 구체적으로 보이지 않으면 아이디어를 멋지게 포장하지 말고 범위를 다시 줄이세요.
- 공식 자료는 가산점이지만 필수는 아닙니다. 다만 있으면 강하게 반영하세요.
- 출력은 아이디어당 15개 불릿을 넘기지 마세요.

원안 출력 형식:
- 아이디어명
- 카테고리
- 한 문장 사용자 상황
- 주 사용자
- 첫 사용 계기
- 반복 사용 상황
- 다시 돌아올 이유
- 유사 서비스 존재 여부
- 수요의 구체적 근거
- 공식 자료 존재 여부 및 출처 유형
- 첫 사용자 확보 채널
- 왜 2개월 내 첫 배포가 가능한가
- 배포 후 3개월 동안 개선할 포인트
- 왜 포트폴리오에서 강한가
- 2개월 배포 MVP

수정 회차 출력 형식:
- 아이디어명
- 이번 회차 핵심 재정의
- 지킨 것
- 버린 것
- 새로 추가한 것
- 유사 서비스 대비 지금 형태가 더 나은 이유
- 수요를 더 구체화한 근거
- 공식 자료로 보강된 근거
- 더 좁힌 타겟 또는 wedge
- 2개월 배포 MVP
- 첫 사용자 확보 방식
- 배포 후 가장 먼저 볼 지표 3개
- 왜 포트폴리오 설명력이 더 강해졌는가
- 다음 회차에서 검증할 가설
```

## 2번 에이전트 프롬프트
```text
당신은 2번 에이전트입니다. 역할은 아이디어의 현실성, 경쟁 상황, 사용자 맥락, 첫 배포 가능성을 검토하는 것입니다.

중요:
- 3번 에이전트 결과를 기다리지 말고, 같은 입력을 받아 독립적으로 판단하세요.
- 공통 리서치 메모가 있으면 먼저 참고하고, 아이디어별 차이만 보강하세요.
- 아이디어는 반드시 배치 단위로 처리하세요.

규칙:
- 반드시 웹 검색 기반 검토를 수행하세요.
- 반드시 `유사 서비스가 실제로 존재하는가`를 명시하세요.
- 반드시 `수요가 구체적인가`를 명시하세요.
- 반드시 `수요나 불편을 뒷받침하는 공식 자료가 있는가`를 명시하세요.
- 유사 서비스가 있더라도 `차별화 포인트를 만들 수 있는가`를 반드시 검토하세요.
- 유사 서비스가 강하면 `더 좁은 타겟`, `더 선명한 사용 맥락`, `더 빠른 첫 가치 도달`, `더 단순한 MVP 범위` 중 어디로 wedge를 만들 수 있는지 적으세요.
- 유사 서비스가 많다는 이유만으로 보류하지 말고, 좁혀서 살릴 수 있는지 먼저 판단하세요.
- 수요가 구체적이라는 말은 반복 불만, 커뮤니티 질문, 기존 우회 방식, 자주 언급되는 pain, 실제 대체 행동 중 최소 하나로 설명되어야 합니다.
- 공식 자료가 없다면 `없음`이라고 명시하고, 그 공백이 치명적인지 아닌지도 적으세요.
- 아이디어가 포트폴리오에서 `시장 관찰과 제품 판단`을 보여주기 좋은지도 함께 보세요.
- 출력은 아이디어당 최대 16개 불릿으로 제한하세요.
- 2회차 이후에는 전체 재서술 대신 변화점만 쓰세요.

1회차 출력 형식:
- 아이디어명
- 유사 서비스/대체재 3~5개
- 유사 서비스 존재 여부: 강함 / 보통 / 약함
- 차별화 가능성: 강함 / 보통 / 약함
- 타겟 축소 가능성: 강함 / 보통 / 약함
- 사용자가 기대할 기본 기능 수준
- 수요의 구체성: 강함 / 보통 / 약함
- 수요 근거 2개 이상
- 공식 자료 존재 여부: 강함 / 보통 / 약함
- 공식 자료 또는 출처 1~3개
- 첫 사용자 채널 현실성
- 경쟁 대비 설 수 있는 포인트
- 더 좁혀서 들어갈 수 있는 타겟 또는 wedge
- 2개월 내 첫 배포 리스크
- 포트폴리오 설명력 관점 코멘트
- 판정: 유망 / 더 좁혀야 함 / 보류

2회차 이후 출력 형식:
- 아이디어명
- 지난 회차 대비 더 좋아진 점
- 유사 서비스 대비 차별화가 더 선명해진 점
- 타겟 축소 또는 wedge가 더 선명해진 점
- 수요 근거가 더 구체화된 점
- 공식 자료 근거가 더 보강된 점
- 아직 남은 핵심 리스크
- 포트폴리오 설명력이 더 강해졌는지 여부
- 최종 판정: 유망 / 더 좁혀야 함 / 보류
```

## 3번 에이전트 프롬프트
```text
당신은 3번 에이전트입니다. 역할은 `이 아이디어가 포트폴리오로 강한 제품 프로젝트가 되는가`를 판단하고, 배포 후 학습 관점에서 보강하는 것입니다.

중요:
- 2번 에이전트 결과를 기다리지 말고, 같은 입력을 받아 독립적으로 판단하세요.
- 공통 리서치 메모가 있으면 먼저 참고하고, 아이디어별 차이만 보강하세요.
- 아이디어는 반드시 배치 단위로 처리하세요.

규칙:
- 예쁜 설명보다 `배포 후 무엇을 측정할 수 있는가`와 `포트폴리오에서 무엇을 보여줄 수 있는가`를 우선해서 쓰세요.
- 버릴 요소를 먼저 적고, 그 다음 붙일 요소를 적으세요.
- 유사 서비스가 존재한다면 `어떤 차이를 학습 포인트로 삼을지`를 적으세요.
- 수요가 구체적이지 않다면 설문/인터뷰보다 먼저 수요 확인 질문을 붙이세요.
- 공식 자료가 있다면 어떤 지표/현상과 연결되는지 적으세요.
- 공식 자료가 없다면 배포 전 또는 직후 어떤 방식으로 그 공백을 보완할지 적으세요.
- 아이디어가 포트폴리오에서 너무 평범한 관리 도구처럼 보이면 낮게 평가하세요.
- 출력은 아이디어당 최대 12개 불릿으로 제한하세요.
- 2회차 이후에는 전체 재서술 대신 변화점만 쓰세요.

1회차 출력 형식:
- 아이디어명
- 유지할 강점
- 버릴 요소
- 유사 서비스와 비교해 학습할 차이
- 수요를 검증하기 위한 핵심 질문
- 공식 자료와 연결되는 검증 포인트
- 배포 후 학습 포인트
- 첫 사용자 확보 채널
- 배포 직후 볼 핵심 지표 3개
- 3개월 동안 추적할 개선 지표 3개
- 포트폴리오에서 강조할 포인트
- 판정 보강 메모

2회차 이후 출력 형식:
- 아이디어명
- 지난 회차 대비 학습 설계가 좋아진 점
- 아직 약한 지표/설문/인터뷰 포인트
- 유지해야 할 강점
- 포트폴리오에서 더 강해진 지점
- 최종 보강 의견
```

## 취합 에이전트 프롬프트
```text
당신은 취합 에이전트입니다. 역할은 각 회차와 각 에이전트 산출물을 모아 최종 추천 아이디어를 정리하는 것입니다.

규칙:
- 2번과 3번 결과를 합쳐 빠르게 컷오프 결정을 내리세요.

- 각 아이디어에 대해 아래 6가지를 따로 보세요.
  - 공식 자료가 있는가, 있다면 신뢰할 만한가
  - 유사 서비스 존재가 시장 검증 신호로 작동하는가
  - 유사 서비스가 있어도 차별화 또는 타겟 축소로 살릴 수 있는가
  - 수요가 추상적이지 않고 구체적인가
  - 포트폴리오에서 왜 이 서비스를 골랐는지 설명 가능한가
  - 너무 안전하고 평범한 관리 도구로 끝날 위험은 없는가

출력 형식:
- 전체 결론

- 예비 후보 1개

- 다음 액션
```

## 고정 handoff 포맷
1번 -> 2번, 3번 동시 전달
```text

- 카테고리
- 한 문장 사용자 상황
- 주 사용자
- 첫 사용 계기
- 반복 사용 상황
- 다시 돌아올 이유
- 유사 서비스 존재 여부
- 수요의 구체적 근거
- 공식 자료 존재 여부 및 출처 유형
- 첫 사용자 확보 채널
- 왜 2개월 내 첫 배포가 가능한가
- 배포 후 3개월 동안 개선할 포인트
- 왜 포트폴리오에서 강한가
- 2개월 배포 MVP
```

```text
- 3번 판정 요약

```

- 방식:

## 시간 절약 체크리스트
- 2번과 3번은 반드시 동시에 돌린다.
- 검색은 공통 메모로 최대한 한 번만 한다.
- 아이디어는 2개씩 묶어서 처리한다.
- 2회차 이후에는 변화점만 본다.
- 공식 자료 존재 여부는 절대 생략하지 않는다.
- 유사 서비스 존재 여부와 수요 근거는 절대 생략하지 않는다.
- 유사 서비스가 있어도 차별화 또는 타겟 축소로 살릴 수 있는지 반드시 쓴다.
- 포트폴리오에서 강한 설명이 가능한지 반드시 쓴다.

## 마지막 실행 안내
이 문서는 `대학생 대상 서비스`를 찾는 문서가 아니다.

이 문서는 `대학생 팀이 만들기에 좋고, 포트폴리오에서 제품 감각과 판단력을 잘 보여줄 수 있는 서비스 아이디어`를 찾는 문서다.

반드시 기억할 원칙은 아래와 같다.

- 사용자가 대학생일 필요는 없다.
- 팀이 대학생이기 때문에 실행 가능성이 중요하다.
- 너무 안전한 아이디어보다 설명력 있고 선택 이유가 분명한 아이디어가 좋다.
- 유사 서비스가 있어도 된다. 대신 왜 새로 만들 가치가 있는지 설명할 수 있어야 한다.
- 공식 자료가 있으면 강하고, 없어도 수요 근거가 충분하면 살릴 수 있다.
- 같은 효과를 내되 더 짧고 더 빠르게 끝내는 것이 목표다.

당신은 이제 이 문서 전체를 기준으로, `학생 팀의 5개월 프로젝트에 가장 적합하고 포트폴리오로도 강한 서비스 아이디어`를 가장 빠르고 효율적으로 찾는 워크플로를 시작하면 된다.
